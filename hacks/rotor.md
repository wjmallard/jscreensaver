# rotor — port notes

Port of `rotor.c` — "Draws a line segment moving along a complex spiraling curve." A short, colour-cycling trail chases the tip of a chain of pivoting arms whose lengths and speeds drift forever. See [[squiral]] for the shared skeleton; this follows the vector-trail idiom of [[xspirograph]] and the colour handling of [[braid]].

Original: <https://www.jwz.org/xscreensaver/> · source: `xscreensaver-6.15/hacks/rotor.c` (~394 lines). Attributed to Tom Lawrence (1997) in the man page/xml; the code descends from xlockmore (Patrick J. Naughton, 1991; put into xlock by Steve Zellers, 1990). Removed from the XScreenSaver distribution as of 5.08.

## Algorithm
`count` arms are summed tip-to-tip into one endpoint, starting at screen centre. Arm *e* contributes `(cos, sin)(gAngle * ratio_e) * radius_e`. Each arm's `ratio` (angular frequency) and `radius` (length) linearly interpolate from a start value to a random end value over a random number of frames (`drift_max` ∈ [10000, 110000)); when a counter reaches its max it latches the end as the new start and re-rolls a fresh target, so the figure morphs continuously and never repeats. The global angle `gAngle` oscillates in `[0, MAXANGLE=3000]`, winding **up** at +0.01/frame and **back down** at −0.1/frame (10× faster), so the endpoint sweeps out along the curve and retraces. The last `cycles` endpoints live in a ring buffer; the trail is drawn as a polyline whose colour advances one palette step per frame.

## Rendering
The trail is short (`cycles` points, default 20), so each step the canvas is cleared and the whole ring buffer is redrawn — `nsave-1` one-segment strokes, each in the smooth-colormap colour stored for its far endpoint. Radii are scaled by `devicePixelRatio` (S) so the figure keeps its size on retina; line width is scaled by S too.

## Palette
`rotor.c` is an xlockmore hack with `#define SMOOTH_COLORS`, so its colours come from `make_smooth_colormap` — a per-run random loop through 2–5 HSV anchors, usually muted (sage green, teal, dusky blue, olive… re-rolled each run), **not** a vivid full-saturation rainbow. The port builds it with `makeSmoothColormapRGB(ncolors)` (shared `colormap.js`, a faithful port of `utils/colors.c`). `MI_NPIXELS == ncolors` (200 by default; verified live with `-ncolors 12`, which yields a 12-colour loop), so the segment colour `pix` advances one index per step and the short trail shows a *sliding slice* of the loop — sometimes a narrow band, sometimes spanning many hues depending on where the slice sits on the anchor polygon. `ncolors <= 2` is the C's mono path → white. Verified against live captures: the live binary shows the same muted, per-run-varying smooth palette (earlier ports' fixed `hsl(h,100%,55%)` rainbow was the systemic palette bug).

## Timing
Stock `*delay` is **10000 µs** (`rotor.c` DEFAULTS / `rotor.man`). The live binary's delay is a sleep floor; each frame adds a fixed framework overhead. Measured against the `-fps` overlay in the delay-bound regime (Load ~42 %, small/medium figures): **~57.5 fps**, so `OVERHEAD = round(1e6/57.5) − 10000 = 7391 µs` and the loop paces at `(delay + OVERHEAD)/1000 ≈ 17.4 ms`/step. The port measures **~57.6 steps/s**, matching the live rate. (Large screen-spanning figures drop the *live* binary to 11–26 fps — but that is XQuartz software-rendering of long lines stalling the process at *low* CPU load, an artifact neither the original X11 nor this GPU-accelerated canvas shares, so it is not the calibration target.) The port previously shipped a by-eye `delay` of 20000 (≈50 steps/s).

## Deviations from the C
- **No XOR / black-erase; full redraw instead.** The C keeps a persistent window and each frame draws the one newest segment in colour and re-draws the one oldest segment in **black** to erase it (anti-aliasing is explicitly turned off, so the erase is exact). Canvas has no XOR raster op and *does* anti-alias, so a black-over-old-segment erase would leave grey ghosts (a known issue in this repo — see the perf memory). Instead we clear to black and redraw the entire short trail every step. The set of visible pixels is identical to the C's; only the draw method differs.
- **Pre-roll.** `init()` runs the simulation `nsave` times before the first paint so the very first frame already shows a complete trail, rather than building it up over the first ~20 frames (the C starts from an all-centre buffer). This nudges `gAngle` by ~`nsave`×0.01 ≈ 0.2 and the drift counters by `nsave` — negligible against drift maxima of 10000+.
- **`count` minimum 1, not 0.** The man page's range starts at 0, which in the C `calloc(0)`s and draws nothing (endpoint stuck at centre). The slider floors at 1. The CLI-only "negative count = random arm count" feature isn't reachable from the UI and is omitted.
- **Retina line width.** The C doubles `linewidth` when the window is wider/taller than 2560 px as a manual retina hack; we apply the real `devicePixelRatio` (S) instead and drop the ×2, so thickness tracks the actual display density.
- **Line caps.** The C uses `CapButt`; we use butt for thin lines but `round` for thick ones (>3 logical px) so the gaps between separately-stroked segments don't show — same idiom as [[braid]]. Cosmetic only.
- **The `refresh_rotor` / `redrawing` path is omitted** — it lives under `#ifndef STANDALONE`, so it is never compiled into the screensaver build and `redrawing` is always 0.

## Correctness self-review
- **No closure/termination to get wrong.** Unlike spirograph-type hacks there is no figure that must "close" — the motion runs forever, so there is no exact-float-equality reset that could fail to fire and no dead-line state. The only state machine is the `forward` flip at the two angle bounds (0 and 3000), and **both** branches re-seed direction and clamp the angle, so it can't get stuck at an endpoint.
- **Every drift re-target re-seeds what the next frame reads** (`start`, `end`, `drift_max`, `drift_now`) exactly as the C does; the seed values (`drift_max = drift_now = 1`, `end_radius = 100`, `end_ratio = 10`) make the first step trigger a re-target so the figure starts from the same state as `init_rotor`.
- **Divergence-free.** Verified with a 200k-step headless harness: no NaN/Inf, endpoint stays within a few hundred px of centre (range ≈ screen-sized), max per-step jump ≈ 226 device px (during a fast retrace), `gAngle` stays in `[0, 3000]`. `cos`/`sin` of large arguments (up to ~30000 rad) are finite in double precision — no fixed-point/`>>` overflow exists in this hack.
- **Pause/resume** resets `lastTime = 0` so resume never bursts; **reinit** clears to black and re-seeds a fresh figure.

## Config
Units/defaults mirror `rotor.c`'s DEFAULTS and `rotor.man`: `delay` (µs/step, **10000** stock, *Frame rate*, inverted), `cycles` (trail length / ring size, 20, *Length*), `count` (arms, 4), `size` (line thickness; <0 = random 1..|size|, −6), `ncolors` (200). `delay` is **live** and maps 1:1 to the stock resource; the OVERHEAD (see Timing) is added only in the pacing math. `cycles`/`count`/`size`/`ncolors` size buffers or the palette, so changing them re-runs `init()` via `reinit()`. The micro sign in the unit string is the `µ` escape, never a literal byte.
