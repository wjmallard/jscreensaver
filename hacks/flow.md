# flow — port notes

Port of `flow.c` by Tim Auckland (1996–2004), incorporating code from Stephen Davies — "flow of strange bees": a swarm of particles flowing through the phase space of a 3D strange attractor, viewed through a tumbling/riding camera, with a background search that discovers brand-new attractors. *"Strange attractors formed of flows in a 3D differential equation phase space."*

Original: <https://www.jwz.org/xscreensaver/> · source: `xscreensaver-6.15/hacks/flow.c` (~1216 lines)

## Algorithm
Each step picks (at init) one of five standard attractors — **Lorentz**, **Rossler**, **RosslerCone**, **Birkhoff** (periodic), **Duffing** (periodic) — encoded as coefficients of a generic 3D cubic polynomial ODE (or a sinusoidally-forced periodic variant when `y` is a wrapping time axis). Every "bee" is a point that integrates that ODE with **2nd-order Runge-Kutta** (`Iterate`, step `0.04`), aging a short tail each frame so it leaves a trail. A `discover()` routine flies a pair of test bees ~6000 steps to measure the attractor's **bounding box** (`mid`/`size`) and Lyapunov exponent; that frames the camera and tunes the step. The camera either **orbits** the box on a Lissajous path or **rides** bee 0 (with bee 1 as a "wing" pinned along the up-axis so roll is defined), interpolating toward the target over `chasetime` frames; an orientation matrix is built from the camera head/tail/wing and used to project every tail point with a perspective divide. An optional **bounding box** (36 edges) is 3D-frustum-clipped before projection. Bees that fly past `LOST_IN_SPACE` (2000) are disabled; if all are lost, or after `cycles` steps, the mode re-seeds with a fresh attractor. With `search` on, every frame also tests a random polynomial flow and promotes any genuinely strange one it finds.

## Module shape
`start(canvas) -> { stop, pause, resume, reinit, config, params }` — see `squiral.md`. Mirrors the attractor-iteration idiom of `thornbird.js` and the many-moving-particles structure of `grav.js`.

## Rendering — sparse vector, full repaint
flow.c's default (and the only path on macOS/jwxyz) is **double-buffered**: clear the whole buffer each frame and re-draw every segment. This port does exactly that — `step()` projects all bee trails into one `Path2D`-style segment list **per colour bucket** (`col = bee % (ncolors-1)`, colour `col+1`, like the C's `MI_PIXEL`), and `render()` clears to black and strokes each bucket once. So a few hundred/thousand short segments cost only `<= ncolors` strokes. Simulation (`step()`) and painting (`render()`) are split so `render()` runs once per display frame even when the lag-accumulator steps the sim less than once per frame (large `delay`) — the image stays stable, no flicker. `<= 2` colours falls back to mono white, as the C does.

## Loop
Standard rAF lag-accumulator paced by `config.delay` (µs → ms), 8-step catch-up cap (see `squiral.md`). **Framerate calibration (`OVERHEAD = 12779`):** `delay` defaults to the **stock 10000 µs** (1:1 with the xml); the loop adds `OVERHEAD` so `(delay + OVERHEAD)` reproduces the live binary's effective rate — the live `-fps` overlay shows flow at **43.9 fps**, so `10000 + 12779 = 22779 µs → 43.9 steps/sec`. See the framerate-calibration note.

## Deviations from the C
- **Full-repaint only**: implements the double-buffer path; the non-dbuf "erase old segments individually" path and its `old_segs` bookkeeping are dropped (the result is pixel-identical to dbuf). No X11 GC / `XDrawSegments` machinery.
- **Finite (NaN) guard added**: the C disables a bee only on `fabs(coord) > LOST_IN_SPACE`; since `NaN > x` is false, a NaN bee would slip through. We also reject `!isFinite`, so a single diverging bee can never poison the field (the harness confirms no NaN ever leaks). The same guard is added inside `discover()`.
- **Buffers rebuilt per re-seed**: the C allocates the segment/bee buffers once (sized by the *first* `taillen`), then reuses them even though `taillen` is re-randomised on every `init_flow` (a latent under-size risk). We rebuild the bee tails each `init()` to the current `taillen`/`count`, and only rebuild the colour buckets when `ncolors` changes (so the last frame of the old flow survives one frame into a re-seed, matching the dbuf "draw old frame, then reinit" timing).
- **Config mirrors `flow.xml`**: `count` (3000, range 10–5000), `cycles` (10000, range 0–800000), `ncolors` (200, range 1–255) and `size` (−10, range −20…−2) use the stock defaults and xml ranges. `delay` uses the **stock 10000** too; the rAF loop adds `OVERHEAD` to reach the live binary's effective rate rather than raising the default (the slider still maps 1:1 to the xml; see the Loop section). `count` is a sparse-vector load (≤ `ncolors` strokes/frame over a few thousand short segments), so the stock 3000 renders fine; if retina stroke cost (lineWidth 3) ever janks, the slider reaches down to 10.
- **Colours** *(fixed in the fidelity audit — was the systemic Rule-1 bug)*: flow.c defines **none** of `UNIFORM_COLORS`/`SMOOTH_COLORS`/`BRIGHT_COLORS`, so xscreensaver's xlockmore layer gives it the **default** colour scheme — `make_random_colormap` with `bright_p = FALSE` (`xlockmore.c`): `ncolors` **fully-random** RGB entries (each channel an independent random 16-bit value), **not** a smooth or ordered rainbow (so some entries are muted/dark, exactly like the original). Ported faithfully via `colormap.js`'s `makeRandomColormapRGB(ncolors, false)`. The colormap is built **once** (xlockmore allocates colors at screen setup; `init_flow` never rebuilds them), so we cache it and re-roll only when the `ncolors` slider changes — an internal timeout / all-gone re-seed keeps the same palette, as the C does. Bucket `col` uses palette entry `col+1` (the C's `MI_PIXEL(col+1)`); `≤ 2` colours falls back to mono white (`MI_WHITE_PIXEL`). *(Previously a vivid HSL rainbow `hsl(h,100%,55%)`.)*
- **devicePixelRatio**: backing store sized in device px; `lineWidth` follows flow.c's own retina rule (`3` past 2560 px, else `1`) — since the canvas is already device-px this matches the C directly. `CapNotLast`/`JoinMiter` line attributes are X11-specific and irrelevant to canvas (defaults used).
- **`Gauss_Rand` w==0 guard**: added (the C omits it) to avoid `log(0)`; measure-zero, no behavioural change.
- **`--fps`/`--root`/`-dbuf` omitted**: X-specific (as in the other ports). `rotate`/`ride`/`periodic` are non-live (they shape the camera/attractor → `reinit()`); `box`/`search`/`delay`/`cycles` are live.
- **`size` (Length of trails)**: kept faithful to the xml — a `-20..-2` slider with `convert="invert"`, which the C turns into a *random* `taillen` (`n²+MINTRAIL`, re-rolled each re-seed). So trail length varies per attractor, exactly as upstream; the slider value shows the raw negative number, as xscreensaver's own UI does.

## Correctness self-review
- **Divergence**: every bee is checked for non-finite/`>2000` coords *before* it is iterated or drawn; a lost bee is frozen (never re-iterated) so it can't propagate garbage. The camera bee, if lost, hops onto a random surviving bee (the C's anti-reinit trick). The periodic `y` axis is wrapped each frame (`y -= 2π`) both in the head and across the tail, so it can't grow unboundedly.
- **Termination/reset**: two reset conditions both re-seed a fresh attractor — `swarm == 0` (all bees lost, guarded by `count > 1` so it can't fire on the empty first frames) and `count++ > cycles` (timeout). `chasetime` is floored at 1 (never divides by zero) and counts down to 1, guaranteeing the camera reaches its target in finite steps. `nbuckets = max(1, ncolors-1)` so `bee % nbuckets` never mods by zero (the C's `% (NPIXELS-1)` divides by zero at `ncolors==1`).
- **First frame**: with `box` on (default), frame 1 already draws the bounding box; bees become visible within ~3 steps (the C ramps `end = min(taillen, count)` from 0). `discover()` runs at init so `size`/`mid` are valid before the first projection, and the orientation matrix is non-degenerate on frame 1 (forward ≠ side for the orbit camera). Confirmed by hand.
- **Numeric harness** (`flow_harness.mjs`, throwaway in the scratchpad — mirrors `cubic`/`periodic`/`iterate`/`discover` verbatim): ran all 5 attractors, 200 bees × 1000 steps each, plus 8 `discover()` framings each. Result: **ALL MODES PASS** — `discover()` frames a finite, positive, in-range `size`/`mid`; **no NaN ever leaks**; the bulk of bees survive bounded (Lorentz/Duffing 200/200, Rossler/RosslerCone 199/200, Birkhoff ~156/200 — the rest escape and are disabled, exactly the C's documented "bees fly away at the start"), with `maxAbs` 6–43 (far under the 2000 escape radius).

## Things to spot-check in the browser
- Camera transitions: occasional orbit↔ride switches (`search` finds a keeper, or the random `NRAND(1000)/4000` rolls) should glide over ~100 frames, not jump.
- `search` runs a full `discover()` every frame (~4–5 M float ops/frame); if it ever janks on a weak machine, the `Search for new attractors` checkbox (live) disables it, and attractor variety still comes from the `cycles` timeout re-seed.
- The periodic attractors (Birkhoff/Duffing) should read as a Poincaré-slice "bagel", not streak in Y — the tail-hiding on the `y` wrap handles that.
