# kumppa — port notes

Port of `kumppa.c` by Teemu Suutari (1998) — "Spiraling, spinning, and very, very fast splashes of color rush toward the screen." Fresh color is injected near the center every step, then the *entire framebuffer* is spun outward about the center, so every mark gets pulled into a spiral streak racing off the edges. The spin is a faithful transcription of the C's fixed-size block shear (a size-preserving translation per block — **not** a zoom). See [[squiral]] for the shared skeleton this follows.

Original: <https://www.jwz.org/xscreensaver/> · source: `xscreensaver-6.15/hacks/kumppa.c` (~545 lines, much of it the elaborate integer rotation tables).

## Algorithm
Two looks, chosen by the `random` resource:
- **cosilines ON (default)** — eight oscillators with fixed phase increments and amplitudes (the C's `cosinus[8][6]` table) drive four wandering line endpoints. Each step every line is drawn from its previous endpoint to its new one, cycling through the color ramp. The result is smooth Lissajous ribbons.
- **cosilines OFF** — eight random small colored squares are splatted within ±16px of the center each step (`Satnum` placement, `fgc[Satnum(50)]` color, with indices ≥32 falling back to the black background, so some splats punch holes).

Either way a small black square is stamped dead-center (`fgc[32]`, the background) so the very core never bakes to a solid blob, and then the framebuffer is spun+zoomed by `rotate()`.

The `rotate()` / `make_rots()` / `palaRotate()` machinery in the C is a hand-rolled, fixed-point pixel-shuffle. `make_rots()` precomputes, via a void-and-cluster-style dither, which screen columns/rows fall in which of the `rotsizeX = (int)(2/speed + 1)` groups (one group is consumed per frame, so `rotsizeX` is also the cycle length). `rotate()` then `XCopyArea`s the window onto itself as a grid of **fixed-size blocks**, each displaced by an integer `(du-dv, du+dv)` shear about `(midx,midy)` (`du,dv` = the block's signed distance from center) — a spiral motion (rotation + outward drift) assembled from size-preserving block *translations*, never a pixel scale. Higher `speed` = fewer/larger groups = bigger shear per step = faster outward streaking. The xml mislabels `speed` as **"Density"**; it is really the spin rate.

## Shared skeleton (inherited)
Standalone ES module exporting `title` + `start(canvas) → { stop, pause, resume, reinit, config, params }`; rAF **lag-accumulator** loop at fixed timestep (`config.delay` µs) replacing the C's `usleep`; `devicePixelRatio` folded into a `pscale` (mark/line size) and the backing store sized in device px; vivid HSL ramp; descriptive names.

## Deviations from the C
- **Feedback: faithful block-copy, transcribed verbatim (the load-bearing one).** `make_rots()`, `rotate()`, and `palaRotate()` are ported line-for-line from `kumppa.c` with the exact integer arithmetic (`Math.trunc` matches C's `(int)` cast because both operate on IEEE-754 doubles — e.g. `(int)(2/0.1+1) = 21` in both). The **only** substitution is the copy primitive: the C's `XCopyArea` becomes `ctx.drawImage(scratch, ax,ay,w,h, dcx,dcy,w,h)` with **equal src/dst size** and `imageSmoothingEnabled = false` — a pure, size-preserving integer blit, the canvas equivalent of `XCopyArea`. Each frame issues roughly one block-copy per grid cell (≈186 at 320×240, ≈4600 at 1080p, speed 0.1), matching the C's per-frame `XCopyArea` count, and the C's diamond iteration order is preserved.
  - **Double-buffered, not in-place (a browser necessity, still faithful).** The C's default `useDBE` is off, so it `XCopyArea`s `win[0]`→`win[0]` *in place*. Doing that on a canvas (`drawImage(canvas, …)` onto itself) forces a GPU read/write **sync on every one of the thousands of block-copies per frame** — fine for X11's server-side blit, ruinous in a browser (kumppa became unusable on a Retina display). So each step snapshots the frame into a `scratch` canvas once and every block copies **from** it — which is exactly the C's `useDBE`-**on** double-buffer mode (a faithful C variant, not an approximation): blocks read a frozen source instead of partially-written pixels, and the result is visually identical (verified against the live binary).
  - **Resolution cap (the second half of the perf fix).** Double-buffering removed the per-call GPU sync (which had locked the tab), but a single step is still ~one `drawImage` per grid cell and that count scales with W·H — at native Retina res a step is ~12ms on an M4, slower than the 10000µs default interval, so the lag-accumulator death-spirals into multi-step frames (~100ms). The backing store is therefore capped to `MAX_EDGE = 2048` px on the longer edge (CSS-upscaled to fill, the spirit of the C's commented-out `.lowrez: true`), cutting the block count quadratically to ~4–5ms/step so the default interval is comfortable. `MAX_EDGE` is one tunable const (raise for sharpness, lower if laggy; `Infinity` = full device res). The rAF catch-up is also capped at 2 steps so a heavy frame can't burst.
  - **Possible future improvement (WebGL).** The block-copy count is the only real cost, and it is embarrassingly GPU-friendly: a WebGL port could render all ~6,500 blocks as **one batched textured-quad draw** sampling a ping-pong feedback texture (1 GPU call vs thousands of `drawImage`s), removing the resolution cap and running faithfully at full device resolution. A proof-of-concept parallel port lives at `hacks/shelved/kumppa-webgl.js` (not registered, not a replacement for this canvas port). This is the model for any other block-copy/self-feedback hack that hits the same wall.
  - **Why the previous port was wrong (and a scale can never be faithful).** It faked the whole feedback with a single `ctx.drawImage` under `ctx.scale(z)` (`z = 1 + speed/2 ≈ 1.05`). A uniform scale **magnifies the entire frame ~5% every step**, so every coloured line *thickens* ~5%/frame; over the ~60 frames a mark takes to spiral center→edge it balloons ~19×, the lines merge, and the black gaps flood to solid (and read as disjointed). The C never scales — it *translates* fixed-size blocks — so the lines keep their width and the gaps survive. No choice of `z`/smoothing fixes this; the magnification is the defect.
  - **Speed is a live slider; the C fixes it at init.** `make_rots()` runs once per (speed, size). When the Density slider moves we rebuild the tables and reset the cycle phase (`stateX/stateY`) **without clearing the screen** — only the spin rate changes. Build cost is ~1ms at the preview size and ~40-50ms at 1080p, one-off per distinct value.
- **Color ramp.** The C hardcodes a 32-entry ramp (`colors[96]`, kumppa.c:72-79) whose hue **decreases** from blue: 240→180→120→60→0→300→270→240 (blue→cyan→green→yellow→red→magenta→violet→blue). Reproduced as a resizable vivid HSL wheel starting at hue 240 and sweeping a full turn **downward** in that same direction, exposed as `ncolors` for parity with the other ports. kumppa builds its own RGB ramp, not `make_smooth_colormap`, so `colormap.js` is intentionally not used here.
- **Retina scale.** The C triples line width on >2560px displays; folded into `pscale` as a ×1.5 bump (×3 over-dominated the canvas port) on top of the dpr.
- **Encoding.** The micro sign in the Frame-rate slider unit is emitted via a Unicode escape, not a literal byte (no literal non-ASCII in a rendered string); literal Unicode appears only in comments.
- **rAF lag-accumulator vs `usleep`.** The C `usleep`s `delay` µs between steps; the port banks elapsed time and runs fixed `(delay + OVERHEAD)`-µs steps off `requestAnimationFrame`, so the pace matches at any refresh rate. `speed` keeps the stock C/xml value (0.1; kumppa.c:52). **Framerate calibration (`OVERHEAD = 8484`):** `delay` is the stock **10000 µs** (1:1 with the xml), but that is only a sleep floor — the live binary's real rate is lower. The live kumppa measures **54.1 fps**, while the port at the stock delay ran 100 steps/sec (1.85× fast); `10000 + 8484 = 18484 µs → 54.1 steps/sec`, matching the live binary. A calibration, not a tuning knob — see the framerate-calibration note. (The earlier by-eye 20000 µs default is dropped.)

## Correctness self-review
The risk for a self-feedback hack is the loop running away to **all-black** (content collapses to the center) or **flooded** (content saturates). Traced for the block-copy feedback:
- **No black-hole collapse:** every block is *translated outward* — the `(du-dv, du+dv)` shear points away from center and grows with distance — never shrunk, so content marches off the edges and is clipped there; black is replenished at the **center** by the per-step black stamp. Content exits at the rim rather than imploding to a point.
- **No flood/white-out:** the copy is **size-preserving** — a block keeps its exact pixel dimensions — so lines never thicken and the black gaps between the spiral arms persist. The only color *added* per step is the handful of center marks; the outflow at the edges balances that small injection, so coverage settles into a moving steady state. Verified headlessly: replaying the real module's own draw calls (lines + stamps + ~186 block-copies/step) on a CPU pixel model over 200 steps holds the painted fraction at ~8–12% (peak 16.7%), never approaching solid — the gaps survive (the old scale path, by contrast, magnified the gaps shut).
- **State re-seeding:** `reinit()` resets the transform to identity, clears to black, and re-runs `init()`, which rebuilds the rotation tables (`make_rots`), zeroes `acos`/`coords`/`ocoords`/`drawCount`, resets `stateX/stateY`, and rebuilds the palette — a clean fresh screen with no stale oscillator phase. `resize()` does the same and re-derives `cx,cy`/`midx,midy`/`sizx,sizy`/`pscale` for the new size (the C's `kumppa_reshape` resets `midx,midy` and the rotation state; the port additionally rebuilds the tables for the new size, which is strictly safer).
- **Pause/resume:** `pause()` cancels rAF (`rafId = 0` sentinel); `resume()` resets `lastTime = 0` before re-arming so no catch-up burst fires — the spin simply continues from the frozen framebuffer.
- **`ctx` transform hygiene:** the block-copy feedback uses **no** `ctx` transform at all — every `drawImage` is given explicit integer source/destination coordinates — so there is nothing to leak; `resize()`/`reinit()` still `setTransform(1,0,0,1,0,0)` before clearing, keeping the identity transform the mark drawing assumes.
- **Index safety:** `rotateX/rotateY` are `Int32Array(2*(maxi+2))`; across all `rotsizeX × rotsizeY` cycle phases the diamond loop tiles every real grid block exactly once and never indexes past those arrays (verified). The C's few spurious boundary calls (reading past the last breakpoint) are reproduced harmlessly: unset entries read as `0`, and `palaRotate`'s `ax<bx && ay<by` clip rejects them, exactly as the clip rejects the C's uninitialised entries there.
- **Index math:** the C's `((a<<2)+draw_count)&31` color walk is reproduced with a floored modulo over `colors.length` (resizable), so it stays in range for any `ncolors`; the splat path maps clamp-high indices (≥32) to black exactly as the C maps them to `fgc[32]`.

## Config
Mirrors `hacks/config/kumppa.xml`: `delay` (µs/step, default **10000** — the xml stock, `invert:true` Frame-rate slider showing raw µs; the loop adds a measured `OVERHEAD` so the pace matches the live binary, see above), `speed` (0.0001–0.2, default 0.1 — labelled **Density** in the xml; it is the spin rate, `live` — moving it rebuilds the rotation tables in place), `random` (smooth-lines toggle, default on, non-live → `reinit`). `ncolors` (32) is added for parity. `live: true` keys (`delay`, `speed`) are read every step; `live: false` keys (`random`, `ncolors`) size the look/palette so a change re-runs `init()` via `reinit()`.

**Local dev:** ES-module imports mean `file://` double-click won't load it (CORS on the `null` origin). Serve it — `python3 -m http.server`, then <http://localhost:8000/#kumppa>. GitHub Pages serves over http, so production is unaffected.
