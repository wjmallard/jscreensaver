# loop вАФ port notes

Port of `loop.c` by David Bagley (1995), implementing Chris Langton's self-reproducing loops (Langton, *Self-Reproduction in Cellular Automata*, Physica 10D 135-144, 1984).

Original: <https://www.jwz.org/xscreensaver/> ¬Ј source: `xscreensaver-6.15/hacks/loop.c` (~1700 lines, the bulk of it the unported hexagon variant + its rule tables).

Shares squiral's skeleton (inline ES module, `config` object, rAF + lag-accumulator loop, dpr scaling) and is the closest sibling to [`ant.js`](ant.md) вАФ both are Bagley square-grid CAs that draw one colored block per changed cell and reset periodically. See [`squiral.md`](squiral.md), and the grid-CA twins [`demon.md`](demon.md) / [`cloudlife.md`](cloudlife.md) (Uint8 grid, double buffer, colored per-cell block, periodic restart).

## Algorithm
An 8-state CA on a square (von Neumann, 4-neighbour) grid. From a single seed вАФ the 10x10 **Adam loop** вАФ a sheathed strand of "genetic" data circulates around the loop; when it reaches the loop's open construction arm it extrudes a copy, and the daughter then closes into its own loop and starts reproducing. The colony grows outward like a coral reef: the outer loops keep spawning while inner ones, walled in by their own daughters, fall dormant. It restarts when the colony fills the dish (`generation > cycles`) or the pattern stops changing (`dead`).

### The transition table (the crux)
The whole behaviour lives in `TRANSITION_TABLE`, transcribed **verbatim** from loop.c's `transition_table[]` (219 octal rules; audit-verified numerically equal entry-for-entry, as is the 10x10 Adam seed). Each rule is an octal word read right-to-left as `CBLTRI`: low digit `I` = next state, then `R`,`T`,`L`,`B` (the 4 neighbours), high digit `C` = the cell's own (center) state. Rules are packed into a 4096-entry lookup `table[(B<<9)|(L<<6)|(T<<3)|R]`, with all 8 center-state outputs stuffed 3 bits each into one entry вАФ exactly loop.c's `TABLE`/`TABLE_IN`/`TABLE_OUT` macros.

Each rule is inserted under **all four 90-degree rotations** вАФ `(R,T,L,B)`, `(T,L,B,R)`, `(L,B,R,T)`, `(B,R,T,L)` вАФ so a cell's next state is rotation-invariant (depends only on its state and the cyclic order of its neighbours), matching the C's four `TABLE_IN` calls. Neighbour directions match `position_of_neighbor`: `R`=east(0¬∞), `T`=north(90¬∞), `L`=west(180¬∞), `B`=south(270¬∞). The CA has a chosen **handedness**: a random per-restart `clockwise` flag mirrors both the seed (`ADAM[j][N-1-i]`) and the lookup (`TABLE_OUT(c, B,L,T,R)`), per loop.c's `init_adam` / `do_gen`. Only one handedness works for a given table; both mirror images reproduce identically.

## Correctness self-review
The brief's decisive check is whether a seed loop actually **replicates** rather than freezing or dissolving into static noise. Verified offline (headless Node) before shipping:

- **It reproduces.** From the Adam seed (population 72) the live-cell count climbs steadily вАФ 99 @gen50, 200 @gen200, 436 @gen400 вАФ and the number of connected components rises 1 вЖТ 2 вЖТ 3 вЖТ 4 as daughter loops detach. Growing population + multiplying disjoint structures is the signature of correct self-reproduction; a wrong table either dies to 0 or freezes at the seed. (These figures were measured flaw-free; the default `--count -5` now sprinkles 0..5 blue debris per restart, which perturbs the exact counts but not the reproduction.)
- **Both handedness variants replicate** identically (mirror symmetry), confirming the mirror seed + mirror lookup pairing is right.
- **The bounding-box optimisation is exact.** loop.c only scans/steps an active rectangle that grows as the colony spreads. I do the same, but verified it generation-for-generation against a full-grid double-buffer reference: the two grids stay **bit-identical through 600 generations**. The box never clips the wavefront because `initAdam` seeds it one cell larger than the loop footprint and the front advances at most one cell per generation, so the box is always вЙ•1 cell ahead of any change. (Background outside the box can't spontaneously light: the `0000000` rule keeps a 0-cell surrounded by 0-cells at 0.)
- **Restart re-seeds what the next step reads.** `init()` zeroes both buffers, repaints the dish black, drops a fresh Adam loop, and resets `generation` and the active box вАФ so a restart (fill, freeze, or `reinit`) gives a clean screen, and `pause`/`resume` only toggles the rAF (resetting `lastTime` to avoid a catch-up burst).

## Deviations from the C
- **Square grid (4 neighbours) only.** loop.c's default `-neighbors 0` randomises between this square grid and a **hexagonal (6-neighbour)** variant (a 262144-entry `hex_transition_table` + hexagon geometry/rendering); the hex variant вАФ most of the 1700-line file вАФ is **not** ported. The square loop is the iconic, recognisable case. *This is the one substantive fidelity gap: the live binary shows hexagons ~50% of runs.*
- **Palette is now faithful (audit fix).** loop.c `#define`s `UNIFORM_COLORS`, so the framework builds the colormap with `make_uniform_colormap` = `make_color_ramp(0,S,V вЖТ 359,S,V)` with `S` (saturation) and `V` (value) each a single **random value in 66%вАУ100%** held for the whole session (`utils/colors.c`). `init_loop` then assigns the 8 state colours by sampling that hue ramp at `k¬Јncolors/REALCOLORS` (REALCOLORS = 6) for the chromatic states, plus black (0) and white (7). We rebuild the *exact* ramp via `colormap.js`'s `make_color_ramp` and pick the same indices вАФ so the palette is a **per-session random-S/V hue wheel** (often a touch muted), not fixed vivid primaries, and `ncolors` quantises the hues just as the C does. S/V stay fixed across colony restarts (as in the C, where the colormap is built once); they reroll only when `ncolors` changes. *(The previous port used fixed full-saturation HSL with slightly wrong hues and ignored `ncolors` вАФ corrected.)*
  - **Not ported:** the `MI_NPIXELS < 8` *stipple* fallback (1-bit-display monochrome patterns per state). For `ncolors < 8` the port keeps drawing colours rather than stipples вАФ a non-issue on a TrueColor canvas.
- **Blue-wall flaws now ported (audit fix).** loop.c's default `--count -5` calls `init_flaw` `NRAND(6)` (0..5) times per restart, dropping a small 3-cell **blue elbow** of debris at random spots that the colony collides with вАФ added in 2001 "to liven it up" (and to spawn the occasional mutant). The square-branch placement is transcribed verbatim; `--count` is not an xml resource, so it stays at the baked-in `-5`. *(The previous port omitted flaws entirely вАФ corrected.)*
- **Random size + orientation now faithful (audit fix).** `--size` follows loop.c exactly: **negative = random magnitude** in `[MINSIZE=5, |size|]` per restart, `0` = auto-fit, positive = fixed, all capped so вЙ• `MINGRIDSIZE=30` cells fit. The xml default is `-12` вЗТ a random 5вАУ12px cell each restart (the previous port hardcoded a fixed 12px вАФ corrected). The seed is also placed at a **random 90¬∞ orientation** (`dir = NRAND(4)`, via the dirx/diry basis) on top of the random handedness, matching `init_adam`. loop.c's `>2560px` retina `√Ч3` heuristic is replaced by dpr scaling (constant CSS-px cell, sharpened to device px); the `xs вИТ (xs>3)` 1px gridline gutter and the dead 1-cell "petri dish" border ring are preserved.
- **Config units faithful (audit fix):** `delay` is the xml resource in **microseconds** (default 100000 = 100 ms/gen, the stock pace), divided to ms for the rAF clock вАФ not a re-scaled ms knob.
- **Framerate calibration (`OVERHEAD = 12360`).** The stock `delay = 100000 ¬µs` is only a sleep floor; the live binary's real rate is lower (delay + framework overhead). The live `-fps` overlay shows loop at **8.9 fps**, so the loop adds `OVERHEAD`: `100000 + 12360 = 112360 ¬µs вЖТ 8.9 steps/sec`. (The overhead is only ~12% at this slow delay, but it lands cleanly on the standard accumulator.) A calibration, not a tuning knob вАФ the `delay` slider stays 1:1 with the xml. See the framerate-calibration note.
- **Rendering:** sparse `fillRect` per **changed** cell (the wavefronts) plus a one-time full paint of the seeded dish, not a per-frame blit вАФ the active set each generation is small.

### Order-of-operations note (port vs C)
loop.c's `draw_loop` diffs+draws first, *then* `do_gen`s; this port `doGen`s first, then diffs/draws/grows the box. The displayed sequence (seed, gen1, gen2, вА¶) is identical, but because `doGen` runs **before** the box-growing diff, the active box must already carry a 1-cell margin around every live cell вАФ so `init()` pads the seed/flaw footprint by 1 (then clamps inside the border). The wavefront advances вЙ§ 1 cell/gen, so the margin is always sufficient. (Minor consequence: the port shows one extra generation per lifespan вАФ `cycles+1` vs the C's `cycles` вАФ perceptually nil.)

## Config
Mirrors `loop.xml` 1:1: `delay` (Frame rate, ¬µs, inverted, live) ¬Ј `cycles` (Timeout вАФ generations before restart, live) ¬Ј `ncolors` (Number of colors, reinit) ¬Ј `size` (Size, reinit; negative = random magnitude). Handedness **and** 90¬∞ orientation are randomised on every (re)build, so `reinit` / restart can flip the spin direction and rotate the seed.
