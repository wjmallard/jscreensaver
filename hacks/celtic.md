# celtic — port notes

Port of `celtic.c` by Max Froumentin (2005; copyright 2006) — repeatedly draws random Celtic knot-work: an interlaced band woven through a planar graph and drawn as smooth over/under crossing strands.

Original: <https://www.jwz.org/xscreensaver/> · source: `xscreensaver-6.15/hacks/celtic.c` (~1140 lines).

Shares the module skeleton (inline ES module, `config`/`params`, rAF + lag-accumulator loop, dpr scaling) with [[squiral]]; the grid-graph idea is the same family as [[truchet]] and [[penrose]] (graph/grid tiling), but here the graph is only scaffolding — the drawn object is the *knot* woven around it.

## Algorithm
1. **Roll random params** (`rollParams`, faithful to `celtic_draw`'s `switch(random()%4)`): curve/shadow width, `shape1`/`shape2` (the spline bulge), `edge_size`, `margin` (often negative, so the pattern overflows the screen), a rotation `angle`, and one of four **graph types** — `grid`, `kennicott` (5-node diamond clusters), `triangle` (triangular lattice), `polar` (concentric orbits).
2. **Build the graph** (`make*Graph` + `graphRotate`). Each `Edge` stores its two endpoint angles (computed once, then the whole graph is rotated rigidly — the angles go stale but the trace only uses angle *differences*, which are rotation-invariant, so this matches the C exactly).
3. **Trace the bands** (`traceSplines` = `pattern_make_curves`). Each edge has two "couples" — a clockwise and an anticlockwise side. Starting from the first unused couple, the walk repeatedly: marks the current couple used, finds the **next edge around the current node** in the current turn direction (smallest angular step), emits one cubic **Bezier** segment that rounds that node (midpoint→midpoint), then crosses the edge (move to the other node, flip the direction). It closes when it returns to the exact start triple `(node, edge, direction)`. Repeat until every couple is used, producing one closed spline (band) per orbit; 2-segment degenerate orbits are dropped (the C's "just one point" case).
4. **Reveal with over/under** (see below), then **linger** and regenerate (clear → new graph → new palette).

## Rendering — sparse vector strokes, progressive reveal
Knots are thin smooth strands, so this uses **canvas vector ops**, not the per-pixel blit path. The reveal is the C's `pattern_animate` reproduced faithfully: all bands advance in lock-step along a shared parameter `t ∈ [0,1]`. Per tick, for each band, it draws (a) a short **shadow** segment in the background colour (`#000`), a little *ahead* of the tip and wider than the strand, then (b) the short **coloured** tip segment. When `t` reaches 1 a back-redraw repaints each band's seam in colour to cover shadow spillage (the C's final loop). Strands are coloured per band from the smooth colormap (single-band knots instead cycle the colormap per segment, the C's `nb_elements==1` special case). `lineCap`/`lineJoin` are `'round'` (the C's `CapRound`/`JoinRound`).

### Over/under is *emergent* (honest caveat — please spot-check)
The woven look is **not** an explicitly computed crossing assignment. It falls out of the draw order exactly as in the C: at a crossing the band whose `t` is larger arrives later, so its shadow erases the earlier band and its colour paints on top → the earlier band dips under. This produces a convincing weave but does **not** mathematically guarantee strict alternation at every crossing — it is whatever the shadow trick yields, identical to the original. Worth a visual spot-check that crossings look interlaced rather than muddy on the busy `kennicott`/`triangle` knots.

## Deviations from the C
- **Erase transition → instant clear.** The C clears between knots with `erase_window`'s wipe animation; the host owns transitions, so a finished knot **lingers** (`delay2` seconds) then the canvas clears instantly before the next knot. No XOR anywhere (celtic never used it).
- **Adaptive tick budget.** The C runs a fixed 100 ticks per draw-call regardless of band count. Busy knots (`kennicott` averages ~22 bands) would mean thousands of individual canvas strokes per frame, so `nticks = clamp(1200/activeBands, 4, 100)`: simple knots keep the C's ~1 s reveal, busy ones reveal a bit slower but hold the frame rate. The per-tick parameter `STEP` is kept at the C's `0.0001` (bands hold up to ~9700 cubic segments — measured — so a coarser step would draw visibly polygonal chords). Shadow look-ahead loops are capped (`guard < 1000`) purely defensively.
- **Colour (audit fix).** The C builds a `make_smooth_colormap` (a coherent 2–5 anchor-hue loop, re-rolled each knot in both `celtic_init` and the per-knot recolor block of `celtic_draw`) and reserves indices 0/1 for fg/bg; the bands only ever index 2..ncolors-1 (`random()%(ncolors-2)+2`, and `segment%(ncolors-3)+2` for the single-band case). The port now uses the shared **`makeSmoothColormapRGB(ncolors)`** (a faithful port of `utils/colors.c`) with the same index arithmetic, re-rolled every knot — so the palette is the original's muted, coherent smooth colormap, **not** the vivid full-saturation `hsl()` rainbow the port used before (the systemic audit bug). Indices 0/1 are unused by the bands, so no fg/bg special-casing is needed. Mono path dropped.
- **`devicePixelRatio`.** `curve_width`, `shadow_width`, `edge_size`, `cluster_size`, and `margin` are all scaled by `S = dpr` so the knot is the same physical size and the over/under gap (driven by `shadow_width`) reads correctly on retina. The graph is built directly in device px (`canvas.width/height` as the C's window size).
- **Random type each knot (no steering knob).** The C re-rolls the graph type every knot (`NRAND(4)`); celtic.xml exposes no type resource, so the port always rolls random too — an earlier draft added a `Pattern` selector, removed 2026-07-01 as invented (Rule 3). The two integer quirks in the C's roll are preserved verbatim, including the famous `random()%1*2-1.0` (which is *always* `-1.0`) in the grid branch, and the triangle graph's transposed create-vs-edge indexing.
- **Speed / units (audit fix).** `delay` is in µs at the **xml stock default 10000** (was an eased-by-eye 50000 — removed). The loop now paces at **(delay + OVERHEAD)**, not raw delay, because each `celtic_draw` also spends real time drawing (up to 100 ticks of short segments per band), so the live binary's true frame period is delay + draw cost — the reveal must run at that cadence, never faster than the author's floor. `OVERHEAD = 7800` µs is live-measured (live 56.1 fps mid-reveal, Load 43.9%, clean at stock delay 10000). `delay2` is the linger in seconds at the **xml stock default 5** (was an eased **3** — removed): the finished knot holds *statically* for `delay2` s (no drawing), then the canvas clears and a fresh knot is rolled. Keypress / `fps` handling dropped (host owns keys and the meter).

## Correctness self-review
A headless harness (`scratchpad/harness.mjs`) drove **4000** random knots across all four types, stubbing canvas/window and exercising `rollParams`/`buildGraph`/`traceSplines`/`splineValueAt`:

- **Bands always close, and the trace is a clean partition.** Independently re-walking the directed edge-side permutation, **every** `(edge, direction)` couple belonged to exactly one closed orbit that returned to its start triple — `coverage failures: 0`, with `coveredCouples == 2 * edgeCount` on every trial. So no band runs forever and none is left half-traced (the do-while geometric closure on object identity — not float equality — is what the C uses, ported verbatim). The trace also carries a defensive `2*E+10` cap that never fired.
- **Counts match the C's intent.** `grid` produced **exactly 2** bands every time (the C's even-rounding comment), `kennicott` ~22, `triangle` ~9.5, `polar` ~1.7 — i.e. real multi-strand knots, not a single trivial loop.
- **No NaN / Infinity.** Sampling every band at 100 points: `non-finite splines: 0`. The `nsteps`/`os` denominators are guarded (`>= 1`).
- **No freeze / no runaway.** `0 throws`. The reveal terminates deterministically at `t >= 1` (then lingers by wall-clock and regenerates), so it cannot wedge. The lag accumulator caps catch-up at `MAX_CATCHUP_STEPS = 2` (each step is stroke-heavy). `pause()`/`resume()` use the `rafId === 0` sentinel and reset `lastTime` to avoid a burst; `reinit()` clears to black and re-seeds.

**Spot-check in the browser:** confirm (1) the strands look **interlaced** (over/under) rather than flat-overlapping, especially on `kennicott`/`triangle`; (2) each band reads as a continuous **closed** loop with no broken seam; (3) the pattern overflows the screen edges (negative margin) and rotates at a random angle; (4) after a knot completes it holds briefly then a **new** type/palette appears; (5) toggling **Draw graph** overlays the scaffold.
