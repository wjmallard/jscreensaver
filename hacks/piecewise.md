# piecewise — port notes

Port of `piecewise.c` by Geoffrey Irving (2003) — a swarm of circles drifts and bounces around the screen, intersecting each other; in the original, a circle's outline flips between visible and invisible at every crossing point. Here they are drawn as translucent filled discs that blend brightly where they overlap. See [[squiral]] for the shared skeleton; closest technique twins are [[braid]] (filled arcs) and [[greynetic]] (alpha compositing).

Original: <https://www.jwz.org/xscreensaver/> · source: `xscreensaver-6.15/hacks/piecewise.c` (~1035 lines)

## Algorithm
The C is a small computational-geometry engine. Each frame it runs a Bentley-Ottmann **plane sweep** (a top-down **splay tree** for the event queue and another for the active "fringe" of circle boundaries — ~600 of the 1035 lines are the splay code and the sweep) to find every point where two circle boundaries cross. Walking the boundary of each circle, its visibility is XOR-toggled at each intersection it passes through, so where circles overlap the arc inverts — hence "switch from visibility to invisibility at intersection points." Geometry: `count` circles (default 32), each with a radius drawn from `[minradius, maxradius] * screen_height`, a random in-bounds start, and a velocity `(1 + rand·0.5) · speed/10` at a random angle. `move_circle` advances them and reflects off the four walls. A single `draw_gc` colour walks a 256-entry rainbow loop, advancing one step every `100/colorspeed` frames, so the whole picture slowly shifts hue.

## What this port keeps faithfully
- **Motion**: `initCircles` / `moveCircle` are a direct transcription — same radius band (`r0 = ceil(minradius·H)`, `dr = floor(maxradius·H) − r0 + 1`), same in-bounds seeding (`r + frand(W−1−2r)`), same velocity formula and random heading, same wall reflection with clamp-back. Circles are inside the field and bouncing from frame one.
- **Colour cycling cadence**: `colorIterations = colorspeed ? max(1, ⌊100/colorspeed⌋) : 100000`, advance the shared `colorIndex` by 1 (mod `ncolors`) every that-many steps — exactly the C's `color_iterations` logic (including the "≈never" branch when colorspeed is 0).
- **Counts/sizes/defaults**: `count` 32 (4–100), `minradius` 0.05, `maxradius` 0.2, `ncolors` 256, `colorspeed` 10, `speed` 15, `delay` 10000 µs — all 1:1 with `piecewise.xml`.

## Deviations from the C
- **The whole plane-sweep is gone.** Reproducing the splay-tree arrangement + per-arc XOR-visibility in canvas would be a large, fragile undertaking for a screensaver, and the per-spawn-prompt guidance is to port the *intent* pragmatically. Instead each circle is a single **translucent filled disc** (`ctx.arc` + `fill` at `alpha`%), composited with `globalCompositeOperation = 'lighter'`. Overlaps therefore **sum to a brighter, more saturated colour** rather than inverting an outline — the readable web analogue of the C's "crossings change the picture." The blend canvas does for free what the C's hand-rolled span compositing did by hand.
- **Outlines → filled discs.** The C strokes arc outlines (`XDrawArcs`); we fill. Filled-and-blended reads better than thin inverted outlines at any DPI and is what the spawn prompt asked for.
- **Per-circle hue offset (added).** The C uses one global colour for all arcs in a frame; that would make every disc identical and overlaps invisible under additive blending. Each circle instead carries a fixed random offset around the wheel, displayed as `(colorIndex + offset) mod ncolors`. The global `colorIndex` still slowly cycles (faithful), but the discs are now distinctly coloured so crossings actually show. This is the one intentional aesthetic departure.
- **`alpha` and `fade` params (added).** `alpha` (default 35%) sets per-disc opacity — lower = more "blend" feel. `fade` (default 0) replaces the hard per-frame clear with a translucent black veil for optional motion trails; at 0 it's a clean clear like the C (the C never trails). Neither exists in the original; both are web conveniences and are documented in the config-box.
- **No double buffering / no XOR raster op.** The canvas is already double-buffered by the browser, so the C's `doubleBuffer` / DBE path is dropped. Canvas has no `GXxor`; the additive blend above is the chosen stand-in (noted because the *original mechanism* is visibility XOR, even though the C draws solid, not XORed, pixels).
- **devicePixelRatio**: radii are computed in device px (`minradius·H` with `H` already in device px) and velocity is scaled by `S`, so size and drift speed look identical on retina; the backing store is sized in device px per the shared convention.

## Correctness self-review
- **No termination / closure state to get wrong.** Unlike braid (which regenerates a figure) or the figure-sweep hacks, piecewise has no per-figure reset — it's a steady-state simulation. There is no "dead lines" or "endless over-draw" failure mode here: every step clears, draws all `count` discs, and moves them. The only loops are bounded `for i < count` passes.
- **Circles stay on-screen.** `moveCircle` reflects *and* clamps (`c.x = c.r` / `c.x = W−1−c.r`) on every wall hit, so even if a velocity overshoots, position is snapped back inside before the next draw — a circle can never drift off the field or get stuck oscillating outside the wall. Seeding (`r + frand(W−1−2r)`) guarantees a fully-inside start, so frame one is already correct. Verified by hand against `move_circle`/`init_circles` in the C.
- **Colour cadence can't divide-by-zero or freeze.** `colorIterations` is `max(1, …)` when colorspeed > 0 and a large constant (100000) when colorspeed is 0, mirroring the C's guard; `colorIndex` is always `mod ncolors` with `ncolors ≥ 2`.
- **rAF loop**: standard shared lag-accumulator with `MAX_CATCHUP_STEPS = 8` and an 8-step lag cap, so a backgrounded tab won't burst on refocus; `delayMs = 0` (max frame rate) is bounded by the step counter, not by `lag`. `pause()` nulls `rafId`; `resume()` resets `lastTime = 0` to avoid a catch-up jump. `reinit()` resets the composite op to `source-over`, clears to black, and re-seeds — a clean fresh screen. (Composite op is also reset in `resize`/`reinit` so a mid-frame interruption can't leave it stuck on `'lighter'`.)
- **Live vs reinit**: `delay`, `colorspeed`, `speed`, `alpha`, `fade` are `live` (read each step); `count`, `minradius`, `maxradius`, `ncolors` size the circle set / hue wheel, so they re-run `init()` via `reinit()`.

## Encoding
ASCII-safe per the project rule: the only non-ASCII in DOM-bound strings is the micro sign in the "Frame rate" unit, written `'µs'`. Em dashes appear only in comments. Verified with `grep -nP "[^\x00-\x7F]" hacks/piecewise.js` (all hits on comment lines).
