# munch — port notes

Port of `munch.c` (Tim Showalter 1997; Jamie Zawinski and Steven Hazel; the `mismunch` variant merged by jwz in 2008) — the classic PDP-1 **"munching squares"** of HAKMEM item 146/147 (MIT AI Memo 239, 1972), discovered by Jackson Wright on the RLE PDP-1 circa 1962. Plot the graph `y = x XOR t` for consecutive values of `t` (= time) and you get nested, animating triangles that "munch" across a power-of-two square.

Original: <https://www.jwz.org/xscreensaver/> · source: `xscreensaver-6.15/hacks/munch.c` (461 lines), config `hacks/config/munch.xml`.

## Algorithm
A **muncher** is one power-of-two square at a random spot. Sizes come from `calcLogWidths()` (== the C's `calc_logwidths`): pick `logMaxWidth = floor(log2(0.8 * smaller_window_dim))` so a square never exceeds 80% of the window (bigger reads as noise), clamp to ≥2, and always offer three sizes `[logMaxWidth-2 .. logMaxWidth]`. Each muncher also gets random wrap offsets `kX, kT, kY`, a `grav` bit (vertical flip), a hand-picked colour (one of four `random()%4` schemes), an optional near-overlapping **shadow** copy (mismunch only), a random starting `y`, and a random **`doom`** value — the `t` at which it aborts (`doom == width-1` draws the whole square).

Each **turn**, `munch(m)` draws one value of `t`: for `x` in `0..width-1`,
- classic: `y = ((x ^ ((t + kT) % width)) + kY) % width`
- mismunch: `y = ((-y ^ ((-t + kT) % width)) + kY) % width` (feeds `y` back into itself with sign flips, so parts of some squares land off-place)

then `drawX = ((x + kX) % width) + atX`, `drawY = grav ? y + atY : atY + width-1 - y`, plot `(drawX,drawY)` (and the shadow). After the `x` loop, `t++`; when `t > doom` the muncher is **done** and replaced.

Several munchers run at once (`simul`). Like the C's `munch_draw`, each frame runs **5 turns** round-robin over the munchers (advancing `drawI`, wrapping at `simul`); when `clear` munchers have completed since the last wipe, the screen clears to black and a fresh batch is made (re-rolling the `random` mode each wipe).

## Shared skeleton
Same as every port — see [[squiral]] for the full description: a no-build ES module exporting `title` + `info` + `start(canvas) → { stop, pause, resume, reinit, config, params }`, an rAF **lag-accumulator** loop in place of the C's `usleep(delay)` (identical pace at any refresh rate, with a catch-up cap so a backgrounded tab can't burst), `devicePixelRatio` folded into the backing-store size, and the config box driven by the inline `params` schema via `config-box.js`. Closest technique twin: [[binaryring]] (Uint32 `ImageData` blit — manual pixel writes + one `putImageData` per frame), with the XOR-emulation idea borrowed from [[halo]].

## Deviations from the C
- **XOR GC → manual per-pixel XOR into a Uint32 buffer (the headline deviation).** The C draws every point through an X11 graphics context set to `GXxor` (default `*xor: True`), so a point laid over an earlier one **toggles** it and overlapping squares **cancel** — that XOR interleaving *is* the munching-squares look. Canvas has no XOR raster op and no XOR GC. Following the brief, this port computes the pattern **directly** into a persistent `Uint32Array` over `ImageData` and XORs each point's packed RGB into the buffer pixel by hand: `pixels[idx] = (pixels[idx] ^ (color & 0x00FFFFFF)) | 0xFF000000` (alpha forced opaque so it never toggles). Because JS bitwise ops are 32-bit two's-complement, this is **bit-identical** to X11 pixel-XOR for the packed 24-bit colours — exact cancel-on-overlap, no antialiasing to muddy it (we write raw pixels, never canvas vector ops). `globalCompositeOperation = 'difference'` was rejected: it is only XOR-equivalent for pure 0/255 channels (halo's case), but munch XORs arbitrary colours, where `difference` ≠ `xor`. Solid mode (`--no-xor`) overwrites the pixel instead, matching the C's `GXcopy` path.
- **`mismunch` recurrence replicated bit-for-bit, NOT renormalised.** The C lets `m->y` go **negative** in mismunch mode (truncating `%`, signed `^`) and relies on `XDrawPoint` clipping the resulting off-window points — that "creative breakage" is the whole point of the variant. JS bitwise ops (ToInt32, two's-complement) match C `int` `^`, and JS `%` truncates toward zero like C `%`, so the recurrence is reproduced exactly with **no** `((n%m)+m)%m` flooring; `plot()` clips out-of-bounds points just as X11 clips to the window. (Flooring `y` into `[0,width)` was tried first and reverted — it changes the mismunch appearance.)
- **Colour packing.** The C's `XColor` channels are 16-bit (`random()%65536` etc.) and go through `XAllocColor`; we pack `>> 8` to 8-bit RGB for the pixel buffer, preserving the four hand-picked colour schemes' balance. XORing two such packed colours is exactly what X11 pixel-XOR of the allocated pixels would do for a direct-colour visual.
- **No `XGetWindowAttributes` per call / no colormap plumbing.** The C re-queries window size and re-allocs the colour inside `munch()`; the canvas size is fixed between resizes and colours are precomputed, so those drop out.
- **Mode select shape.** The xml exposes `mismunch` as a 3-way select (`random` / `munch` / `mismunch`) and the XOR/Solid choice as a second select; here XOR is a checkbox (default **on**, == the C's `*xor: True`). `mismunch` stays a select. `delay` default is the stock **10000 µs** (matches the xml/C; previously 20000 here — restored in the 2026-06-27 audit). Ranges/labels otherwise map 1:1 to `munch.xml`: `clear`/"Duration" 1–200 (65), `simul` 1–20 (5).
- **Framerate calibration (`OVERHEAD = 6667`).** The rAF lag-accumulator reproduces the C's `delay` (one 5-turn `munch_draw` per `delay`), but the stock `delay` is only a sleep floor — the live binary's *effective* rate is lower (delay + per-frame overhead — see the framerate-calibration note). The `-fps` overlay measures **60.0 fps**, while the port at the bare stock 10000 µs ran 100 steps/sec (1.67× fast), so the loop adds `OVERHEAD`: `10000 + 6667 = 16667 µs → 60.0 steps/sec`, matching the live binary. A calibration, not a tuning knob; the `delay` slider still maps 1:1 to the xml. (Note: 60.0 fps reads like it may be vsync-capped on the live host — the main session should spot-verify the visible pace.)
- **Encoding:** the µs unit in the Frame-rate slider is the escape `µs` (ASCII-safe rule); the only non-ASCII bytes in `munch.js` are em dashes and a couple of µ in **comments**.

## Correctness self-review
The brief warns parse-clean + right-contract isn't enough. Traced by hand:
- **Termination / replacement.** Every muncher carries a finite `doom` (`< width` for mismunch, `= width-1` for classic), and `t` strictly increments each turn, so every muncher reaches `done` in at most `width` turns and is replaced — no muncher can run forever, no turn is skipped. The `done` muncher is replaced **in place** by a fresh `makeMuncher()` (every field reseeded: width, position, offsets, colour, shadow, `y`, `t=0`, `doom`, `done=false`), so the next turn that lands on that slot reads a valid square — there is no half-initialised state.
- **Wipe cadence.** `drawN` counts completed munchers; when `drawI` wraps and `config.clear && drawN >= config.clear`, the batch is fully rebuilt, `pixels.fill(BLACK)` wipes the buffer, and `drawN` resets to 0 — so the screen can't grow without bound, and the `random` mode is re-rolled each wipe exactly as the C does. `clear` is **live** (read each frame), so dragging "Duration" takes effect at the next wipe without a reinit.
- **No off-screen / degenerate first frame.** `init()` seeds `simul` munchers and clears the buffer to black before the first frame, and each muncher's `atX/atY` keep its square on-screen (the `W <= width ? 1 : W - width` guard avoids `random()%0`). The very first turn already plots on-screen bands. `calcLogWidths` clamps `logMinWidth ≥ 2` so `width ≥ 4` — no zero-width square, no `1 << huge`.
- **No NaN / no signed surprises.** All arithmetic is integer; `ilog2` floors its arg first (matching the C's truncate-to-`size_t`) and never sees 0 here (dim·0.8 ≥ 1 on any real canvas). It returns `31 - Math.clz32(x)` (highest set bit) to mirror `i_log2`'s clz-based integer op **exactly** — the earlier `Math.floor(Math.log2(x))` could round an exact `2^k` (e.g. `window*0.8 == 1024`) down to `k-1` and pick squares one size too small; fixed in the 2026-06-27 audit. Negative `drawX/drawY` are dropped by `plot()`'s bounds check, not wrapped.
- **pause / resume / reinit.** `pause()` cancels rAF and parks `rafId = 0`; `resume()` resets `lastTime = 0` before re-arming, so no catch-up burst. `reinit()` re-runs `init()` (fresh buffer + munchers, cleared to black) for a clean screen after a non-live change (`simul`, `mismunch`, `xor`). The lag accumulator caps catch-up at `MAX_CATCHUP_STEPS = 8`.

**Spot-check in the browser:** (1) with **XOR on** (default), confirm overlapping squares *cancel* (you see the classic interleaved-triangle "munch", with dark holes where colours coincide) rather than solid stacked blocks — that's the manual-XOR path doing its job; (2) toggle **XOR off** → squares should over-paint (no cancellation), confirming the solid branch; (3) **Mismunch only** should look blockier / broken (off-place fragments, occasional shadow copies) vs **Munch only**'s clean nested triangles; (4) leave it running past `clear` munchers and confirm the screen wipes and regrows rather than freezing or piling up forever.
