# anemotaxis — port notes

Port of `anemotaxis.c` by Eugene Balkovski (2004) — an "optimal search" simulation: odor **sources** on the left emit particles that drift right while random-walking in y (turbulent wind), and **searchers** enter from the right edge and crawl left to find a source, fanning out in expanding zig-zag "cones" until they sense a particle, then heading straight up the plume. Models how moths track pheromone plumes (odor-modulated anemotaxis).

Original: <https://www.jwz.org/xscreensaver/> · source: `xscreensaver-6.15/hacks/anemotaxis.c` (~759 lines).

## Algorithm
- **Lattice**: x in [0, `distance`], y in [0, 2·`distance`], mapped to device px by `X()`/`Y()` (the C's macros, `ax = W/distance`, `ay = H/(2·distance)`). `distance` is the "size of the lattice" (default 40).
- **Sources** sit near the left. Each holds a per-cell array `yvV/yvY` (velocity in {-1,0,1}, `2` = no particle; and the particle's relative y). `evolveSource` drifts every particle one cell outward (random-walking y by ±1) and probabilistically emits a new particle at cell 0 (rate `1/inv_rate`). A source whose emission is silenced (`inv_rate = 0`) is reaped once its last particle drifts away.
- **Searchers** carry a position, a cone `vertex`, a `state` (`UP_LEFT`/`UP_RIGHT`/`LEFT`/`RIGHT`/`DONE`), and a `hist` trajectory. `getV` tells a searcher whether a particle is at its exact cell and, if so, which way the wind blew it (`vel`). `moveSearcher`: if it senses a particle it steps one cell left and shifts y by `-vel` (climbing the plume), re-anchoring the cone; otherwise it walks the expanding triangular cone (up-left, sweep right to the diagonal, up-right, sweep left to the diagonal, …). Landing exactly on a source ends the search with a **white flash** (and silences the source); running off the left edge also ends it. A `DONE` searcher is reaped and respawned at the right edge.
- **Rendering** is a **full repaint each step** (the C clears its back buffer every draw): source blobs as filled discs (size grows with emission rate), drifting particles as small jittered discs, and each searcher's whole trajectory as one polyline (head → history, newest first) capped by a 4×4 square head marker. No persistent canvas / ring buffer — trails live in the history list. Contrast `grav`/`whirlwindwarp`, which keep a persistent canvas and erase-old/draw-new. See [[squiral]] for the shared skeleton, [[grav]] / [[whirlwindwarp]] for the particle-trail idioms.

## Fidelity / palette
This is a close transcription of the .c. `evolveSource`, `getV`, `newSearcher`, `moveSearcher`, the per-slot source/searcher spawn probabilities, the `X()/Y()` mapping, the `dx/dy` jitter, the `dx/3 + 1` line width and the source-blob radius are all copied line-for-line.

- **Palette is faithful.** `anemotaxis_init` builds its colours with `make_random_colormap(bright_p=True)` (utils/colors.c): `ncolors` **independent random** colours — random hue 0–360, saturation 30–100%, value 66–100% (vivid but not full-saturation, often slightly pastel) — built **once** and **never cycled** (the call passes `writable_pP = NULL`, so on a TrueColor visual nothing rotates). The port uses the shared `makeRandomColormapRGB(ncolors, true)` from `colormap.js`, which reproduces that exact HSV distribution, built once in `init()`. Each source/searcher then picks `colors[RND(ncolors)]`; the success flash is white (the C's `WhitePixel`). An earlier port used a vivid evenly-spaced HSL rainbow — the systemic "vivid rainbow over the muted original" bug — now removed.
- **`ncolors` is fixed at 21, not a slider.** The C reads `--colors` (default 20) then does `ncolors++` (a spare cell), so the colormap holds 21 entries. `anemotaxis.xml` exposes **no** colour control (unlike moire/swirl/grav/squiral, whose xml *do* have `<number id="ncolors">`), so the port doesn't expose one either; it's an internal constant.
- **Boots like the C.** `init` creates a **single** source and **zero** searchers, exactly as `anemotaxis_init` (`st->source[0] = new_source(st)`, the searcher array memset to 0). The field then fills in over ~20–40 s through the C's own spawn probabilities, so the port tracks the live binary at every point in time (warm-up included). An earlier port pre-seeded 5 evolved plumes plus a full searcher set walked partway in — removed, since it desynced the port from the live binary at t=0 and produced a non-representative (pure-cone, not plume-following) opening.
- **Spawn rates are the C's.** Sources spawn into a free slot with probability `1/(maxDist·maxSrc)`; searchers with `1/(maxDist·maxSearcher)` — both verbatim. The searcher formula makes the *total* searcher spawn rate near-independent of the slider, so the slider's effect on the steady-state searcher count is sublinear/saturating; this is the stock behaviour and is intentional. The drifting plumes (sources accumulate toward `maxSrc` and rarely die) carry the visual density, so the field is rich once warmed up even though active searchers stay few (~5–12 at the defaults). An earlier port substituted `1/(maxDist·4)` to make the slider scale the population — removed.
- **Disc shapes/sizes are the C's.** Particles and source blobs are circles (`XFillArc`); the searcher head is a fixed **4×4** square (`XFillRectangle`, no Retina bump). Particle disc diameter is the C's `4`, bumped to `8` when the backing store exceeds 2560 px in either axis — the C's exact `(scrWidth > 2560 || scrHeight > 2560 ? 8 : 4)` gate. An earlier port drew square particles sized `4·dpr` (a dpr-scaling deviation, and the wrong shape) — removed.

## Deviations from the C
- **Off-edge retirement hoisted out.** In the C the "ran off the left edge → `DONE`" test lives *inside* the per-source loop, so it is only reached while iterating an active source; with no active source a searcher could wander off-screen indefinitely and grow its history without bound. We test `rx < 0 → DONE` unconditionally (same priority as the C whenever ≥1 source is active, which is the normal case), so a searcher that leaves the field is always retired and history stays bounded. Behaviour is identical to the C whenever any source is active.
- **`delay` default + OVERHEAD calibration.** The default `delay` is the **stock 20000 µs** (the slider maps 1:1 to the xml resource); the loop adds a fixed `OVERHEAD = 7248 µs` so `(delay + OVERHEAD)` reproduces the live binary's *effective* step rate rather than the nominal one. Measured against the live `-fps` overlay anemotaxis runs **36.7 fps**, while the port at the stock delay ran ~50 steps/s (1.36× fast); `20000 + 7248 = 27248 µs → 36.7 steps/s`, matching the binary. A calibration, not a tuning knob — see [[framerate-calibration]]. (An earlier revision used a by-eye default of 50000 µs.)
- **`searchers = 0` mode not ported.** The C has a special mode (sources drift and randomly self-destruct, no searchers; `new_source` also places sources at `x = 0`); the xml slider's minimum is 1, so we always run with ≥1 searcher and omit that branch.
- **Encoding**: the only DOM-bound non-ASCII is the Frame-rate unit, written as the escape `' \u00B5s'`; literal Unicode (em dashes, a µ) appears only in comments.

## Correctness self-review (won't freeze / over-draw)
- **Termination is guaranteed.** A searcher's x strictly decreases on every `UP_LEFT`/`UP_RIGHT`/sense move; the `LEFT`/`RIGHT` sweeps close on **integer-lattice** equalities (`vtx - rx === vty - ry`, etc.) which always fire because the sweep widens by exactly one each cycle. So x reaches `< 0` in bounded time and the searcher is retired and reborn with all fields re-seeded (position, vertex, direction, color, empty history). No searcher can freeze; history is bounded by the off-edge hoist above.
- **No unguarded divides / no float-equality closures.** Screen mapping uses `Math.trunc` (matching the C's `(int)` cast, important since `rx` can be -1); `dx`/`dy` are floored with a `|| 1` floor; the cone closures compare integers. `getV` guards its index against `[0, n)`. `newSource`'s `x = RND(max(1, floor(maxDist/3)))` guards `RND(0)` (the C's `RND(max_dist/3)` is safe only because `max_dist ≥ 11`).
- **Headless smoke (4000 frames at the defaults, plus min/max edge configs):** every draw coordinate stayed finite and every disc radius positive; the per-frame searcher-head/stroke count tracked the faithful sparse equilibrium (~5/frame averaged over the from-zero warm-up); `distance=250 / sources=100 / searchers=100` reinits + runs in single-digit ms/100-frames with no hang; pause yields no pending frame, resume re-banks `lastTime` before its first tick, reinit clears + re-seeds immediately.

## Config
Units/defaults mirror `hacks/config/anemotaxis.xml`: `delay` (µs/step, stock 20000 + an OVERHEAD pacing calibration — see Deviations; `invert` Frame-rate slider, range 0–100000), `distance` (10–250, 40), `sources` (1–100, 25), `searchers` (1–100, 25). The xml exposes no colour control, so the port has no `ncolors` slider (palette size fixed at 21). `delay` is `live`; the rest size the lattice/arrays so changing them re-runs `init()` via `reinit()`.
