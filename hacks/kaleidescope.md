# kaleidescope — port notes

Port of `kaleidescope.c` (Ron Tapia, 1997). The canonical **misspelling** ("kaleidescope", not "kaleidoscope") is preserved in the filename, `title`, and `info` to match the original hack.

Original: <https://www.jwz.org/xscreensaver/> · source: `xscreensaver-6.15/hacks/kaleidescope.c` (~513 lines) · demo video: youtube `mGplFlx1y3M`.

## Algorithm
There are `nsegments` independent line segments (default 7), stored in a "natural" coordinate system centred on the origin. Each step every segment **propagates** (`propigate_ksegment`): its midpoint orbits the screen centre by `global_rotation`, and the segment spins about its own (pre-orbit) midpoint by `local_rotation` (both in units of `2*pi/10000` radians, so the defaults −59 / 1 are small per-step rotations). When drawn, a segment is replicated across an **N-fold rotational symmetry group**: `symmetry` copies, each rotated by an additional `2*pi/symmetry` about the centre (the C's `NEWX/NEWY` macros, applied **iteratively** so copy `k` is rotated `k+1` times), then translated by `(xoff, yoff) = (width/2, height/2)`. Each segment keeps a **ring** of its last `ntrails` positions (the C's circular list of `Ksegment` nodes), so the figure leaves a kaleidoscopic ribbon. If a segment's two endpoints collapse to within `sqrt(100)=10` px, that one node is re-seeded with fresh random endpoints (`draw_ksegment`'s length check → `init_ksegment`).

## Colour — faithful solid-per-segment muted tones (NOT a rainbow)
The default `color_mode` is **"nice"**. Under "nice" the C's `krandom_color` picks each channel **independently at random in `[min, min+range)`** of the 16-bit colour space — the `redmin/redrange/...` resources, all defaulting to `30000 / 20000`, i.e. each channel ∈ `[30000, 50000)` → `>>8` → **`[117, 195)` in 8-bit**. So every colour is a **muted mid-tone**: no channel can reach 0 or 255, so nothing is ever fully saturated or vivid. `kcycle_color` — the per-node trail fade — is gated on `if (! g->color_mode)`, so it is a **no-op in "nice" mode**: every node of a segment shares the **one** colour picked once at creation, and that colour never changes for the run (even a collapse-reset leaves it alone). The trail is therefore a **solid-colour ribbon**, with at most `nsegments` (7) distinct muted colours on screen.

The port reproduces this exactly: `segColor()` = `(nrand(range)+min) >> 8` per channel, one fixed colour per segment (packed into the framebuffer's little-endian RGBA word); no fade, no time-cycling. The six colour-range resources are **not** in the xml's slider set, so they are kept as fixed constants at the .c defaults, not knobs. Headless-capture pure colours land at e.g. (one run) `#76B7B2` (118,183,178), `#B079BE` (176,121,190), `#B4BF83` (180,191,131) — all channels in `[117,195]`.

"greedy" mode (`color_mode==0`), where `kcycle_color` *does* fade each node down to half-brightness across the trail, is not the default and is not ported.

## Rendering — faithful non-AA framebuffer (draw newest / erase oldest)
This is a **direct port of the C's renderer**, not a canvas-stroke approximation. A persistent `Uint32Array` over an `ImageData` (one word per device pixel) **is** the X window: opaque black, and **never cleared** except on init / resize / reset — the C clears the window only at start. The C keeps two X GCs, both with `line_width = lw`, `CapRound`, and **antialiasing OFF** (`jwxyz_XSetAntiAliasing(..., False)`): `draw_gc` (foreground = the node's colour) and `erase_gc` (foreground = background = **black**). Per sim **step**, per object, **in the C's object order**, `draw_ksegment` does exactly two things: **draw** the newest trail node (`cur`) — its `symmetry` copies — with `draw_gc` and set `drawn = 1`; then, if the oldest node (`cur->next`) was ever drawn, **erase** its copies with `erase_gc`. So each step touches only **two line-sets per object** (~`nsegments·symmetry·2` ≈ 154 lines at the defaults); the ~98 middle nodes are **not** redrawn — they persist in the buffer.

The port reproduces this exactly:
- **Non-AA rasteriser.** Each `XDrawSegments` line is drawn by an integer **Bresenham** rasteriser that **hard-sets** each covered pixel to a packed little-endian RGBA word (no blending) — precisely a non-AA X GC. For `lw == 1` (windows ≤ 2560 px) a bare Bresenham line is exact; for `lw > 1` (Retina, `> 2560` px) each Bresenham centre stamps a filled disc of radius `lw/2`, a faithful `CapRound` wide line (the Minkowski sum of the centre line with that disc).
- **Cached screen segments.** When a node is drawn, its `symmetry` copies' final **integer** screen coords `(x1,y1,x2,y2)` — after the iterated, short-truncated `NEWX/NEWY` rotation plus `(xoff,yoff)` — are stored on the node (`seg.xseg`, the C's `xsegments[]`), alongside a `drawn` flag (`seg.drawn`, the C's per-node flag). The **erase** replays those cached integers in black, so it removes exactly the pixels that were drawn.
- **Draw-then-erase order** is the C's: the newest node is drawn *first*, then the oldest is erased, so the black erase **punches gaps** through any still-live line it crosses. That erosion — the churned, cross-hatched texture of the dense centre — is **correct**, not a bug to smooth over.

Because we rasterise non-AA exactly like the C, the port's line edges are **crisp** (hairline, hard-edged), matching the live binary, with **no soft-AA halo**. (The earlier port stroked anti-aliased canvas lines and, because you cannot cleanly erase an AA stroke by overpainting black, had to re-stroke the *whole* ring each step; both problems are gone.)

## Rendering perf — per-step raster, per-frame blit
The raster is off the presentation critical path. Per sim **step** (~30 Hz) it rasterises only the ~154-line delta into the framebuffer, then `putImageData`s the buffer into an **offscreen canvas** once. Every **presentation frame** it `drawImage`s that offscreen 1:1 onto the visible canvas — a cheap GPU blit that also keeps the compositor presenting a live layer (sparse canvas updates otherwise stall the present, the prior finding). So a non-step frame costs just the blit. Measured at 2560×1440 dpr 2 (a 5120×2880 buffer, `lw = 3`): frame-time **p50 ≈ 0.1–0.7 ms, p95 ≈ 1.2–1.3 ms** — well under the 16.7 ms / 60 Hz budget. (The previous offscreen-cache version re-stroked the *whole* ~7,700-line figure on every step; drawing only the two changed line-sets is both faithful **and** cheaper.)

## Deviations from the C
- **Draw/erase GCs ported directly** (see Rendering). Each step the port draws the newest node and erases the oldest into a persistent framebuffer, exactly as the C does — so `ntrails-1` nodes are visible at once (the C keeps exactly one erased). No full-ring redraw, no XOR; the erase is a destructive black overpaint, matching `erase_gc`.
- **No wall-bounce.** The spawning brief described segments "bouncing off bounds", but `kaleidescope.c` has **no bounds-bounce** — motion is purely the global-orbit + local-spin rotations. The only radial wandering comes from **integer roundoff** (the header says so: *"the radial motion of the segments depends on roundoff error alone"*). Endpoints/midpoints are `short int` in the C, so the port truncates each assignment via `trunc16(v) = (v|0)<<16>>16` (round-toward-zero **and** the same 16-bit wrap). With normal window sizes the values stay well inside ±32767, so the wrap never fires in practice; it only guards a runaway excursion, as the C's `short` did. Dropping the truncation makes the figure rigidly rotate with no drift — visibly wrong — so it is load-bearing.
- **`narcs` / `spline.h` dropped.** The C scaffolds an extensible object system (`OBJECT` with `init/draw/propigate` vtables) and reserves an arc/ellipse type, but `narcs` defaults to 0 and no arc object is ever implemented (`create_objects` only makes segments). The port implements just the segment object — identical output — and omits the unused vtable indirection and `spline.h`.
- **`spring_constant` dropped.** Read from resources by the C but never used anywhere (dead). Not exposed.
- **`local_rotation` / `global_rotation` not exposed.** Real `-X` options, but **not** in the xml's slider set, so (like the colour-range resources) they are fixed at the .c defaults (−59 / 1) rather than UI knobs — mirroring the GUI the author shipped. The xml's four sliders (delay / nsegments / symmetry / ntrails) are the only `params`. (The earlier port exposed them as live "Spin"/"Orbit" sliders; removed for xml fidelity.)
- **devicePixelRatio.** Backing store sized in device px; line width is **1 device px, 3 above 2560 px** — a literal port of `kaleidescope_reshape`'s Retina branch (the earlier port used `round(dpr)`). Endpoint magnitudes scale with `xoff/yoff` (= half the device-px canvas), so the figure fills the screen at any DPI. Lines are rasterised **non-AA** (the C's `jwxyz_XSetAntiAliasing(False)`), so edges stay crisp at any DPI — the soft-AA strokes of the earlier canvas port are gone.
- **Pacing.** `delay` is the stock xml value **20000 µs** (the earlier port used a by-feel 16000, and a compensating `local_rotation = −33`; both reverted). No OVERHEAD constant — left for central `-fps` calibration. One `step()` per `delay` via the shared rAF lag-accumulator (catch-up capped at 8 steps), so the pace is the same at any refresh rate.

## Correctness self-review
- **Colour fidelity (the audit fix).** The previous port used a vivid `hsl(h,100%,L)` rainbow with a per-age lightness fade (60→6 %) and a time-cycling `baseHue` — none of which the C does in the default "nice" mode. Replaced with the faithful solid per-segment muted RGB above. Verified by maths (every channel ∈ `[117,195]`, so no vivid colour is representable) and by sampling the headless capture.
- **Reset = one node, not the whole trail.** The collapse check re-seeds **only the live (`cur`) node** (`init_ksegment` touches only `obj->cur`); the rest of the ring is untouched, so the ribbon morphs into its new path over the next `ntrails` steps instead of teleporting. (The earlier port refilled the whole trail on reset, collapsing the ribbon to a point — wrong.)
- **Grow-in.** The C's per-node `drawn` flag drives startup: each step draws one new node but only erases `cur->next` **once that node has been drawn** (`drawn != 0`), so for the first `ntrails` steps nothing is erased and the ring fills in. `done_once` skips the first propagate (the root is drawn in place). No origin-dot cluster ever appears — `propigate_ksegment` writes a node's endpoints **before** it is drawn — so the figure grows smoothly from one segment to the full ribbon over the first ~`ntrails` steps (~2 s at the stock pace). (`nlive` is retained as inert SIM bookkeeping; the render grows in via `drawn`, exactly as the C. The earlier port seeded the whole trail full from frame 1, a different startup.)
- **Symmetry centring.** `NEWX/NEWY` rotate about the origin and `(xoff,yoff)` is added **after** rotating, so all `symmetry` copies share the exact screen centre. The per-copy rotation is the C's iterated, short-truncated macro application (copy `k` rotated `k+1` times), so the small per-step rounding of the original is reproduced rather than a perfect float rotation.
- **Buffers fixed-length.** Per segment the trail ring is `Float64Array(ntrails*4)`, and the render mirrors the C's Ksegment with `drawn` (`Uint8Array(ntrails)`) and `xseg` (`Int32Array(ntrails*symmetry*4)`, the cached screen segments) — all allocated once in `init()`. `cur` wraps mod `ntr`, and the oldest node is `cur->next = (cur + 1) % ntr`. The framebuffer (`Uint32Array` over an `ImageData`) is allocated per `init()`/resize so it always matches the device-px canvas.
- **pause/resume + reinit.** Standard pair (`rafId===0` sentinel, `lastTime=0` on resume to avoid a catch-up burst). `reinit()` → `init()` re-clears to black and re-seeds every segment + ring (fresh colours) — a clean fresh screen.

## Spot-check in the browser
- Colours should be **muted mid-tones** (dusty teal / rose / khaki / lavender), at most `nsegments` of them, each ribbon a **single solid colour** — **not** a bright rainbow and **not** fading along the trail. Each run re-rolls the colours.
- The figure should slowly **drift / wander** outward (radial creep from the `short` roundoff), not just rigidly spin — if it looks perfectly rigid, the `trunc16` truncation regressed.
- It should **grow in** over the first ~2 s from a single symmetric figure to the full dense ribbon (no dot cluster at any point).
- Bumping **Symmetry** / **Segments** / **Trails** (reinit) should change the fold count / ribbon count / trail length and stay centred.

## Left for live verification
- **Pacing** is left at the stock `delay = 20000 µs` with **no OVERHEAD** constant; the true per-step pace wants a live `-fps` reading (central calibration), not a by-feel value.
- **Catalog metadata.** `info.description` was realigned to the verbatim xml `<_description>` (it had been truncated to the first sentence). `catalog.json` is generated from the hack `.js`, so it needs regenerating to pick this up.
