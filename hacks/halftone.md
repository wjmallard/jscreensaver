# halftone — port notes

Port of `halftone.c` by Peter Jaric (2002) — a regular grid of dots whose sizes are driven by the summed gravitational field of several moving point-masses, so the grid ripples and pulses like a halftone print of a slowly churning gravity well. "View it from a distance for best effect."

Original: <https://www.jwz.org/xscreensaver/> · source: `xscreensaver-6.15/hacks/halftone.c` (412 lines). Follows the [[squiral]] skeleton; closest technique twins are [[grav]] (moving point-masses) and [[spiral]] (sparse per-frame draw over a clear).

## Algorithm
- **Masses**: `count` point-masses, each with a position in the normalized 0..1 box, a `mass` in `[minMass, maxMass]`, and a per-axis speed in `[minSpeed, maxSpeed]`. Each step a mass at or past an edge (`pos >= 1 || pos <= 0`) negates that axis's increment, *then* the increment is added — so they bounce inside the box.
- **Field**: the dot grid is `dotsW = floor(width/spacing)+1` by `dotsH = floor(height/spacing)+1`. For each grid node `(x,y)`, `calculateGravity` sums each mass's field as a *vector*: a mass at fraction `(gx,gy)` sits at grid coords `(gx*dotsW, gy*dotsH)`; with `dist` the grid-space distance, each contributes `mass * dotsW * dotsH / dist^2` along the unit direction to the mass. The node's value is the **length of the summed vector** (not the sum of magnitudes — vectors can cancel), clamped to `[0,1]`.
- **Paint**: clear to the background colour, advance the two cycling palette indices every `cycleSpeed` ticks, then draw every node as a filled disc of diameter `sizeFactor * spacing * field` in the foreground colour. Per frame the C does `repaint` then `update` (move masses + recompute field).

## Shared skeleton
Standard for the gallery (see [[squiral]]): a no-build ES module exporting `title` / `info` / `start(canvas) → { stop, pause, resume, reinit, config, params }`; an rAF **lag-accumulator** loop paced by `config.delay` µs (catch-up capped so a backgrounded tab can't burst); `devicePixelRatio` folded into sizes; the host renders the config box from `config`/`params`.

## Deviations from the C
- **No off-screen Pixmap.** The C double-buffers into a `buffer` Pixmap then `XCopyArea`s to the window. The browser already composites the canvas, so we clear + draw straight to it; no flicker. (`update_buffer` and the whole buffer/buffer_gc bookkeeping are dropped.)
- **Retina via `devicePixelRatio`, not a 2560 px threshold.** The C triples `spacing` once the window exceeds 2560 px. We instead scale `spacing` by `devicePixelRatio` (`S`) so dots stay a consistent CSS size and crisp on retina; `maxDotSize = sizeFactor * spacing` follows. `spacing` is floored at 2 px so a dot is always visible.
- **First frame is already filled.** The C's `halftone_draw` paints *before* the first `update`, so its frame 1 shows all-zero (invisible) dots. We call `updateField()` at the end of `init()`, so the very first painted frame already shows the halftone pattern (per the brief's "first frame looks right" mandate).
- **Colors → `hsl()` rainbow.** The C builds a 200-entry *smooth* colormap via `make_smooth_colormap` and uses `color0` (background) and `color1` (= ncolors/2, the dots), each advancing by 1 every `cycleSpeed` ticks. We build an `ncolors`-entry `hsl(h,85%,55%)` rainbow and cycle the same two indices the same way — so the background and dots are complementary hues that slowly rotate. Saturation/lightness are toned slightly below full for a calmer look.
- **Default `delay` 16000 µs** (stock is 10000) — a touch calmer pulse, per the gallery's tuning convention. Everything else (count 10, spacing 14, sizeFactor 1.5, speeds 0.001/0.02, masses 0.001/0.02, ncolors 200, cycleSpeed 10) matches `halftone.xml` exactly. The `cycleSpeed` slider is `live`; it isn't surfaced in the stock xml UI but is a real `--cycle-speed` resource, so it's exposed here (drag right = slower cycling).
- No XOR / feedback raster tricks are involved, so none needed emulation.

## Config
Units/labels mirror `hacks/config/halftone.xml`. `live: true` on `delay` and `cycleSpeed` (read every step → apply instantly). `live: false` on `count`, `spacing`, `sizeFactor`, the four speed/mass sliders, and `ncolors` — they size the grid / masses / palette, so a change re-runs `init()` via `reinit()` (which also clears the canvas). `delay` uses `invert: true` (the xml's `convert="invert"` "Frame rate" slider, drawn right-to-left, showing the raw µs value).

## Correctness self-review
- **No termination / closure state to stall.** This hack has no figure-closing or grid-fill-then-clear state machine (unlike squiral) — every frame it unconditionally clears, moves the masses, and recomputes the whole field. There is no branch that could leave the next frame reading un-reseeded state, so the "dead lines / endless over-draw" failure modes don't apply. The masses move from frame 1 (`moveMasses` runs in every `step`, and the field is already seeded in `init`).
- **Divide-by-zero guarded.** `calculateGravity` only accumulates when `dist !== 0`, exactly as the C does, so a mass landing exactly on a grid node is skipped rather than producing `Infinity`. Positions are doubles seeded by `Math.random()` and advanced by irrational-ish increments, so exact coincidence with an integer node is effectively impossible anyway; the guard is belt-and-suspenders, matching the C.
- **Radius bounded — no blow-up.** The node value is clamped to `[0,1]` before paint, so the disc diameter is bounded by `maxDotSize = sizeFactor * spacing`. At the default `sizeFactor` 1.5 the max diameter is `1.5*spacing > spacing`, so in dark regions dots touch/overlap — this is the intended halftone look and is faithful to the C (which likewise allows `sizeFactor` up to 3). The clamp is what prevents an unbounded "overlap blowout": however close a mass gets to a node, that node's value saturates at 1.
- **Bounce ordering preserved.** The edge test runs against the *current* position before the move (matching the C). Because the speed range is small (≤0.09/step) relative to the box, a mass overshoots an edge by at most one step's increment and is turned back the next frame — it can't escape and stop contributing. `pos <= 0`/`pos >= 1` (not strict) ensures a mass parked exactly on a boundary still flips.
- **pause/resume / reinit.** `pause` cancels the rAF and sets `rafId = 0`; `resume` resets `lastTime = 0` before re-arming, so no catch-up burst and the masses pick up exactly where they paused (no jump). `reinit` re-seeds everything and clears to black for a clean fresh screen.
