# hopalong — port notes

Port of `hopalong` (`hop.c`) by Patrick Naughton (1992), of the xlockmore lineage. Later operations were added by Ed Kubaitis (the EJK1–6 formulas, from xmartin), Renaldo Recuerdo (RR), Clifford Pickover (Popcorn) and Peter de Jong (Jong). Lacy fractal patterns from iterating a strange-attractor map, from a 1986 *Scientific American* article.

Original: <https://www.jwz.org/xscreensaver/> · source: `xscreensaver-6.15/hacks/hopalong.c` (~563 lines)

## Algorithm
The Barry-Martin "hopalong" attractor. A new round picks one of **11 formulas** (Martin/sqrt, EJK1–6, RR, Popcorn, Jong, Sine) and random parameters `a,b,c,d` whose scale is set by `range = sqrt(cx² + cy²) / (1 + rand)`. Then each frame iterates the chosen map `count` times, plotting one point per iteration: most ops do `oldi = i + inc; j = a − i; i = oldj ± f(b·oldi − c)`, then project to screen as `x = cx + (int)(i+j)`, `y = cy − (int)(i−j)` (the cast truncates the **offset** toward zero before adding the integer centre). The points accumulate into a persistent buffer, so a lacy figure builds up over the frames. The whole frame shares **one colour**, advancing one step through a `make_smooth_colormap` palette each frame (so the accumulating image is banded by evolving — often muted — colour, and `count`, the points per frame, controls how many points share a colour, which is why the xml labels it "Color contrast"). After `cycles` frames the buffer clears and a fresh attractor begins.

Special cases match the C: **Popcorn** ignores `range`, reuses `a`/`b` as integer counters and resets `inc` to 100; **Jong** and **Sine** use trig recurrences with their own projections; **EJK6** takes `asin` of the fractional part of `b·oldi` (C's `x − (long)x`, i.e. truncation toward zero → `Math.trunc`).

## Module shape
`start(canvas) -> { stop, pause, resume, reinit, config, params }` — see `squiral.md`. pause/resume use the `rafId === 0` sentinel + `lastTime` reset from `boxfit.js`.

## Rendering — Uint32 blit, accumulating
Point plotting, thousands per frame, accumulating across many frames → the **blit path**: iterate the map writing each point into a persistent `Uint32Array` over an `ImageData`, then `putImageData` once per frame (like `sierpinski` / `binaryring`). Per-point `fillRect` would be thousands of draw calls per frame for no benefit. The C's `XFillRectangles` of a point buffer maps to the same idea; we just keep the buffer between frames instead of redrawing.

## Variable / fixed loop
Fixed-timestep **lag accumulator** paced by `config.delay` (µs, so the loop divides by 1000), with a catch-up cap — identical pace at any refresh rate, no burst on tab refocus. One `step()` = one `draw_hop` = `count` points + one colour advance.

**Framerate calibration (`OVERHEAD = 7271`).** The stock `delay = 10000 µs` is only a sleep floor; the live binary's rate is lower (delay + framework overhead). The `-fps` overlay measures **57.9 fps**, while the port at the bare stock delay ran 100 steps/sec (1.73× fast), so the loop runs at `delay + OVERHEAD = 17271 µs → 57.9 steps/sec`, matching the live binary. A calibration, not a tuning knob; the slider still maps 1:1 to the xml. See [[framerate-calibration]].

## Deviations from the C
- **Faithful port** of all 11 map formulas and their per-op parameter ranges; cryptic names renamed (`i`/`j` → `ii`/`jj`; `pix` kept; `op`/`inc`/`count`→`time` kept close to the C). The per-op recurrences are transcribed line-for-line, including the EJK3/4/5 ternary structure, Martin's `+` vs every other op's `−`, EJK6's `asin` of the truncated fractional part (`x − (long)x`), and RR's signed `pow`.
- **Colour** *(audit fix)*: faithful **`make_smooth_colormap`** (utils/colors.c, via `colormap.js` — 2–5 random HSV anchors interpolated into a loop, *often muted/pastel, not a vivid rainbow*), built **once per `init()`** to mirror the C, which builds it once at startup under `#define SMOOTH_COLORS` (→ `color_scheme_smooth` in `xlockmore.c`) and never rebuilds it. `pix` walks one palette entry per frame, the C's `MI_PIXEL(mi, hp->pix)` advance. When `ncolors ≤ 2` the framework drops to MONO and `draw_hop` skips the colour path, drawing in `MI_WHITE_PIXEL` (`if (MI_NPIXELS(mi) > 2)`); we mirror that with a single white entry (`MI_NPIXELS == ncolors`). *Previously this was a full-saturation HSL rainbow with the .md claiming "house style favours saturated rainbows over the muted X default" — a Rule-1 fidelity violation; removed.*
- **Integer truncation of plotted points** *(audit fix)*: the C casts `centerx + (int)(i + j)` etc. — it truncates the **offset** toward zero *before* adding the (integer) centre. The port now does `cx + Math.trunc(i + j)` (and the analogous Popcorn/Jong forms) per op. Previously it truncated the whole *sum* (`(cx + offset) | 0`), an off-by-one for any negative offset (and, for the subtracted `y`, for positive offsets too) — a 1-px systematic asymmetry across the lacy figure.
- **Integer divisions** *(audit fix)*: Popcorn's `MI_WIDTH/40` and `MI_HEIGHT/40`, and Jong's `4*inc/centerx`, are **integer** divisions in the C (all-int operands). The port now wraps each in `Math.trunc` to match — notably Jong's term stays `0` until `4*inc` exceeds `centerx`, then drifts by small integers, where the old float division gave a smooth sub-pixel drift. Possible because `inc` is now a pure integer (see below).
- **devicePixelRatio**: the backing store is sized in device px and `range` is derived from the device-px centre, so `a,b,c,d` scale with the canvas and the figure fills the screen at any dpr; only the dot SIZE is bumped (`dot = round(dpr)`) for a ~1-CSS-px point on retina (the C uses `scale = 1`, or `3` past 2560 px). `inc` is now left **unscaled** *(audit fix)* — faithful (the C keeps the iteration in raw pixels), it keeps `inc` integer for Jong's integer division, and its per-frame `++` drift dwarfs the tiny initial `[-100,99]` offset anyway. *An earlier port did `inc *= dpr`, which also broke Jong; removed.*
- **Motion is faithful** (this is NOT a static hack): each frame iterates the map `count` times, plotting `count` new points into the **persistent** buffer (accumulating, like the C's un-cleared window) and advancing the colour by one entry; after `cycles` frames the buffer clears and a fresh attractor begins. No colour cycling (the C only rotates a *writable* PseudoColor map; a canvas is TrueColor — cf. `swirl`).
- **Clear-on-restart**: `init_hop` calls `MI_CLEARWINDOW`; we fill the buffer black in `startover()` (no wipes/erase module). As in the C (and `sierpinski`), the clear only shows on the *next* blit, so the finished figure holds for one frame first.
- **Formula select**: the C exposes each formula as a separate `--martin`/`--sine`/… boolean (all default off → `NRAND(OPS)`, fully random). We fold these into one **`Formula` select** (`random` + the 11 named ops) — identical effect (the C picks the first true boolean in fixed order, else random), one control instead of eleven mutually-exclusive checkboxes. The C's `OFFENDING`/`ignoreRotation`/verbose/`XMARTIN` paths are unused.

## Config
Ranges mirror `hacks/config/hopalong.xml`: `delay` (Frame rate, µs, live, inverted, 0–100000), `cycles` (Duration — frames before restart, live), `count` (Color contrast — points/frame, live, 100–10000), `ncolors` (Number of colors, 1–255, reinit), plus the `formula` select (reinit; replaces the xml's eleven mutually-exclusive booleans). Non-live changes start a fresh attractor via `reinit()`. The xml's `showfps` is a framework control (the host shows fps separately).

One deliberate UI clamp (not a fidelity item): the xml's `cycles` range is `0–800000` (default 2500); we cap the slider at `100000` and floor it at `100`, because 0 flashes a new attractor almost every frame and 800000 would pin the default to the far-left of the slider. The map math is otherwise unchanged.
