# vermiculate — port notes

Port of `vermiculate.c` by Tyler Pierce (2001) — "to move in a worm-like manner." A handful of worms crawl across the screen with smoothly turning headings, each laying down a thin coloured trail. The C runs a random one of ~10 sample programs at startup, and about half of them turn `erasing` off, so the port picks one of **two modes at random on each reset**: **slither** (erasing on — fixed-length snakes that stamp a head pixel and erase a tail pixel each step, never saturating the field and never clearing) and **accumulate** (erasing off — the tail is frozen into a *permanent* trail, a worm dies when it wedges, and the field clears + re-seeds once all worms have died). Reload or wait for a reset to sample both.

Original: <https://www.jwz.org/xscreensaver/> · source: `xscreensaver-6.15/hacks/vermiculate.c` (~1228 lines, most of it an interactive scripting layer — see *Deviations*).

See [[squiral]] for the shared module skeleton and [[spiral]] for the circular-trail-buffer idiom this reuses per worm.

## Algorithm
Each worm (`thread`) has a float position `(x,y)`, an integer heading `deg` (0..359), a per-worm grid colour `col`, a turn mode `tmode` (1..7), and the turn-state the mode needs. Every step:

1. **Turn** — the worm's `tmode` mutates `deg`. The seven modes are the meander engine (faithfully transcribed from `move()`): (1) bounded random nudge (±`turnsize`); (2) axis-snapping with occasional sharp turns; (3) constant curvature (steady arc); (4) spiral (the turn-rate itself drifts and reverses); (5) curvy meander — arc one way for ~`curviness` steps, then flip (the classic vermiculate squiggle); (6) alternating straights and arcs; (7) a scripted repeating turn-sequence. Worms are seeded with a random mode so a screenful shows the variety the C's sample strings produce.
2. **Step** — `x += cos[deg]; y += sin[deg]` (exactly one precomputed unit vector, like the C), then wrap toroidally at the edges.
3. **Collide / bounce** — read the 1-px collision grid (`point[]`) at the new cell. If it is inked in a *different* colour (another worm, or the border), reverse (`deg += 180`) and stay put; if it is the worm's *own* colour, pass straight through. (Default worms have `selfbounce`/`realbounce` off; those reflection paths and the grid/killwalls logic are dropped.)
4. **Draw** — stamp a `pscale`-sized rect at the head pixel in the worm's colour `palette[col]`. Once the trail ring is full, handle the oldest recorded pixel per the mode (the C's `move()`, `vermiculate.c:672-678`): **slither** blanks it to black (keeping a fixed-length snake); **accumulate** recolours it to `palette[col + thrmax]` — a *different* random-bright entry — freezing it as a permanent trail. A per-worm circular buffer (`xrec/yrec`, length = a per-worm random `reclen` = 30..199, the C's `newonscreen`) is the moving head. `pscale` is the C's stamp size: **1 device px** normally, 3 on a very large / retina backing store (so the worms are thin — the live binary at a windowed size draws 1-px trails).
5. **Death (accumulate only)** — after recording the head, if the whole `reclen`-long ring has collapsed to a single point the worm is **dead** (it has wedged and can no longer advance): the C's loop-closure scan, `vermiculate.c:683-697`. Dead worms stop moving; their permanent trails remain until the reset.

The worm's colour is *not* decoupled from its grid code: `col = thr + 1` (2, 3, 4, …) is simultaneously the collision marker **and** the colormap index, exactly as the C conflates them (`sp()` sets the pixel and paints `mycolors[col]`). So worm colours are consecutive entries of the palette (see *Palette*), and a worm never shares the border's colour 1 — it always bounces off it.

The **border** (`bordcol`) is colour 1 along the **top and left edges only** — an L-shape, matching the C's `bordcorn==0` (it does *not* draw a full rectangle) — painted in `palette[1]` (the C's `mycolors[1]`, its own random bright colour). Worms that run off the right/bottom wrap toroidally and are caught by this L on the far side. It is **off by default**: nearly every one of the C's sample strings toggles `bordcol` off via a `b`/`B` key, so the live binary almost always shows *no* border. It is exposed as a toggle.

Reset & mode selection: on each reset `doReset()` picks the mode at random (`Math.random() < 0.5` — exactly the C's ~50%, since 5 of the 10 active sample strings turn erasing off). In **slither** mode worms never die (`dead` is only set in the accumulate branch), so the C's `alltrap` reset never fires — **and the periodic `ticks` reset is dead code** (`tick` is a per-call local that never reaches `ticks`=20000, and `had_instring` stays true once the initial sample string is consumed). So slither mode seeds **once** and runs **forever** without clearing (effectively absorbing within a session — reload to re-sample). In **accumulate** mode worms die as they wedge, and once all are trapped `alltrap` fires the reset: the field clears, a **fresh palette is rolled** (the C's `autopal`, which every erasing-off sample sets), and the worms re-seed — after which the mode is re-picked.

## Palette
The C's `randpal()` calls `make_random_colormap(&mycolors[1], &ncolors=tailmax-1=240, …, bright_p=True, …)` — the 6th arg is `bright_p`, and it is `True`, so each of the 240 entries is an **independent random BRIGHT HSV colour**: H random 0–360, S random 30–100%, V random 66–100% (anything from vivid to near-white; the live binary reliably shows one or two near-white worms). It is **not** an even hue ramp. We port it with `makeRandomColormapRGB(240, true)` from `colormap.js` (the faithful port of that exact call), building the whole 240-entry map like the C — accumulate mode needs `mycolors[col + thrmax]` (the frozen-trail colour, index ~122–181) in addition to the worm head `mycolors[col]` and the border `mycolors[1]`. Fixed for a slither session; **re-rolled on each accumulate reset** (the C's `autopal`).

> Earlier this port used an even, full-saturation `hsl(i·360/n, 100%, 55%)` rainbow ramp with a per-worm wheel spread — the systemic vivid-rainbow bug. Fixed to the random bright colormap above; verified against the live binary.

## Shared skeleton
Standard jscreensaver module: `start(canvas) → { stop, pause, resume, reinit, config, params }`, a fixed-timestep rAF **lag accumulator** paced by `config.delay + OVERHEAD` (µs) with a catch-up cap, `devicePixelRatio` folded into the stamp size and the backing store, a `Uint8Array` collision grid, and `wrapAround`/`random1` helpers transcribed from the C's macros.

## Deviations from the C
- **Interactive scripting dropped.** Most of `vermiculate.c` is a tiny keystroke language (`consume_instring`, banks, `pickbank`, grids, borders by corner, prey/follow, killwalls, per-thread bool toggles, the `sampleStrings[]` programs). A screensaver has no keyboard input, so only the **autonomous worm behaviour** the sample configurations produce is ported. What's kept: the 7 turn modes, both trail modes (slither + accumulate) with their death/reset and the autopalette re-roll, the border-and-wrap containment. What's dropped: prey-following (and its `tanof[]`/`atan2` heading code), `realbounce`/`killwalls` wall reflection, the user grid, `little` worms, and all keystroke handling. The firstinit default is `tmode 1` for every worm, but a sample string *always* runs and spreads the tmodes; we seed a random `tmode` (1..7) per worm to reproduce that observed variety rather than a screenful of one mode.
- **Both trail modes ported; selected at random per reset.** The C picks a random one of ~10 sample programs at startup and ~half of them turn `erasing` **off**. Rather than hard-code one, the port picks the mode at random on each reset at the C's rate (`Math.random() < 0.5`, since 5 of the 10 active samples turn erasing off) — no user knob, which would be invented. **Slither** (erasing on): fixed-length snakes, tail erased, no reset. **Accumulate** (erasing off, `vermiculate.c:672-697`): the tail is frozen to `col+thrmax` (a permanent trail), a worm dies once its whole `reclen` ring collapses to one point (it has wedged), and when all worms are dead the field clears, re-rolls the palette (`autopal`), and re-seeds. Since slither never resets it is absorbing within a session; reload (or a fresh accumulate reset) re-samples the mode. Verified: sparse snakes in slither captures, screen-filling permanent geometric/spiral trails in accumulate captures.
- **Rendering: filled rects, not Path2D polylines.** The C draws with `XFillRectangle` (`sp()`), so the port stamps a `pscale`-sized filled rect per head pixel; consecutive one-pixel-apart stamps overlap into a continuous trail. This is *more* faithful to the C than a stroked polyline **and** sidesteps two hazards a stroke-and-erase polyline would have: the long "bridge" segment a worm would draw when it wraps a screen edge, and anti-aliased slivers left behind when erasing. Tail-erase is a black rect at the recorded tail pixel, guarded by re-reading the grid (`gp(ex,ey) === col`) so it never punches a hole in ink a newer head has since laid over that cell (worms bounce off each other's colours, so overlaps are rare and the guard is mostly a no-op).
- **Colour = `palette[col]`.** Faithful: `col = thr+1` indexes the random bright colormap exactly as the C's `mycolors[col]`. Worm colours are consecutive palette entries; the border is `palette[1]`. (Was a vivid even-`hsl()` rainbow — fixed. See *Palette*.)
- **Trail length is per-worm random** `30..199` (the C's `newonscreen`), not a fixed global length.
- **Pacing.** The C's `waitabit` accumulates `threads` per `move()` round and sleeps 10 ms once the sum crosses `speed`, running ~`speed/threads` rounds per frame — so per-worm speed is normalised by the thread count. The port reproduces that: each fixed step runs `round(SPEED_BUDGET / threads)` move-rounds and the accumulator paces at `delay + OVERHEAD` (`delay` = `10000` µs, the C's 10 ms quantum). `SPEED_BUDGET` (240) is the port's analog of the sample string's `speed`, chosen for a faithful ~1500 px/s slither at the default thread count. The xml exposes only "Duration" (the C's `speed`, 1..1000) + fps; the slider is the C's `speed` intent mapped onto the standard inverted **Frame rate** slider. **`OVERHEAD` = 9050 µs, live-measured:** the native binary runs at **52.5 fps (Load 47.5 %, clean)** at `delay 10000`, which `10000 + 9050` reproduces.
- **Config knobs (judgment call).** The xml has *no* resource for worm count, curviness, thickness, colour count or trail length; the C sets them all from the un-portable sample string. The invented **Colors**, **Worm length** and **Thickness** knobs were **dropped** (the palette is always the C's random bright map, trail is per-worm random, thickness = the C's `pscale`). **Worms**, **Curviness** and **Border** are kept (user's call) as adaptation knobs mapping to the C's *interactive* controls (`]`/`[`, `/`/`*`, `B`) at the C's own defaults/ranges (Border now defaults **off**, as the live almost always is).

## Correctness self-review
- **Snake never leaves a ghost trail.** The tail-erase reads `xrec[recpos]/yrec[recpos]` (still the *oldest* pixel — `recpos` is the slot about to be overwritten) before recording the new head, exactly mirroring the C's order. Verified by a headless run: over thousands of steps the count of coloured stamps stays bounded (worms keep a fixed length) and the grid-guard prevents erasing live ink.
- **Worms stay on screen.** Headless bounds check on a 400×300 canvas: with the border **off** (pure toroidal wrap) every stamp falls in `x∈[0,401], y∈[0,301]` (the +1 is the rect width) — no escape, no `NaN` coordinates. With the border **on**, worms draw continuously (alive and bouncing) for thousands of frames.
- **First frame draws.** `init()` sets `resetPending = true`; the first `step()` seeds the worms + border and immediately begins moving/stamping, so frame 1 is already drawing.
- **Reset paths match the C.** Slither: the `ticks` reset is dead code (see *Algorithm*), `alltrap` never fires (worms never die), so it runs once and slithers forever. Accumulate: worms wedge and die, `alltrap` fires the clear/re-roll/re-seed, then the mode is re-picked. Confirmed against captures — sparse thin worms (slither) vs a screen-filling web of permanent geometric/spiral trails (accumulate).
- **No reset state left stale.** `doReset()` re-picks the mode, re-rolls the palette in accumulate mode, and via `newOnScreen()` re-seeds every field the next step reads (`deg`, `recpos`, `turnco`, `turnsize`, `filled`, `dead`, the trail ring), and clears the trail ring so a re-seeded worm can't touch a stale far-away pixel.
- **Pause/resume + reinit.** `resume()` resets `lastTime` so a long pause doesn't fire a catch-up burst. `reinit()` re-runs `init()` (clears to black, rebuilds palette/worms) for a clean fresh screen; non-live params (Worms, Border) route through it, live params (Frame rate, Curviness) are read each step.
- **Edge configs.** Stressed with 60 worms / curviness 50 / 1 worm and border off — no exceptions in any case.

## Config box (shared)
Tunable in-browser via `config-box.js`. `params` declares one entry per key with `live` set: **live** (`delay`, `curviness`) are read every step; **non-live** (`threads`, `border`) re-run `init()` via `reinit()`. `delay` uses `invert: true` (the xml's "Frame rate" intent) and shows the raw µs value.

**Local dev:** ES-module `import`s need a real server (CORS blocks `file://`). `python3 -m http.server` in the repo, then <http://localhost:8000/#vermiculate>.
