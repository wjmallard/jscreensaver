# ifs — port notes

Port of `ifs.c` by Chris Le Sueur & Robby Griffin (2005-2006), after Massimino Pascal's original xscreensaver hack; multi-colour mode by Jack Grahl (2007). Clouds of Iterated Function Systems spin and collide: a few affine "lenses" define the IFS, the chaos game folds a point through them plotting one dot per iteration, and every frame the lenses morph so the fractal writhes.

Original: <https://www.jwz.org/xscreensaver/> · source: `xscreensaver-6.15/hacks/ifs.c` (~559 lines). See [[squiral]] for the shared skeleton, [[swirl]] for the same `make_smooth_colormap` palette helper, and [[thornbird]] / [[hopalong]] for the accumulate-into-a-blit-buffer technique this follows.

## Algorithm
- **Lenses (affine maps).** Each lens has a rotation `r`, scale `s` and translation `tx, ty`. `lensmatrix()` precomputes a 2x2 matrix + offset vector `(ua,ub,uc,ud,utx,uty)` in fixed point: the matrix carries an extra factor of 2^10 and coordinates an extra 2^8, so a step is `(ua*x + ub*y + utx) >> 10` (and likewise for y). `ua = 1024·s·cos r`, `ub = -1024·s·sin r`, `uc = -ub`, `ud = ua`; the offsets centre and scale the figure to the canvas. The six coefficients are stored as integers, exactly as the C's `int ua..uty`.
- **Chaos game (`iterate`).** Repeatedly pick a random lens and apply it; drop the first 10 iterations (burn-in onto the attractor), then plot every later point. The running point persists across passes and frames (the C's `st->x/st->y`, zeroed at init) — it is *not* reset to the centre each frame; the per-call burn-in keeps it on the attractor regardless. Points per pass = functions^(detail-1); "Detail" is exponential.
- **Multi mode (default).** One pass per lens. Pass `i` plots in its own palette colour and applies one extra fixed transform `lens[i]` before plotting, fanning the cloud into per-lens sub-clouds. The dedupe board is reset **before every pass** (the C `memset`s it per lens), so where two passes overlap the later/higher-index pass overwrites — last-pass-wins, exactly as the C; each pass also draws its full cloud rather than skipping pixels an earlier pass already claimed. The base colour index advances each frame so the palette slowly cycles.
- **Morph (`mutate`).** Every frame, each lens eases toward fresh random targets: rotation and scale interpolate sinusoidally (a counter `+= 0.01` per frame, re-rolling a new target when it wraps past 1 — about a 100-frame cycle); translation drifts on a wandering velocity `txa/tya` that is randomly nudged and pulled back when `|tx|,|ty|` exceed 6 or the velocity exceeds 0.05. Then `lensmatrix()` recomputes the fixed-point coefficients.
- **Dedupe + plot (`sp`).** Coordinates arrive in 256ths of a pixel; `>>8` gives the pixel. A per-pass bit board (`getdot/setdot`) ensures each pixel is painted at most once per pass, and out-of-range points are simply dropped (which doubles as the divergence guard). Points are drawn as a `pscale`-sized dot (1px, or 3px on >2560px retina displays, matching the C).

## Palette
`make_smooth_colormap` (the exact `utils/colors.c` routine, via the shared `colormap.js`): 2-5 random HSV anchors interpolated into a closed loop, with min-separation and min-average-saturation/value retries, so it is **often muted/pastel — not a vivid full-spectrum rainbow**. ifs.c builds it **once** in `ifs_init` and never rebuilds it; `ifs_draw` only advances the colour *index* (`ccolour++`, and `ccolour*(i+1)` per lens), so the colours slowly drift through that one fixed map each frame. The port matches: `buildPalette()` runs once per `init()`, floored to at least `functions` colours like the C.

## Rendering
Tens of thousands of points per frame (3^8 ≈ 6.5k × functions passes ≈ 20k+ at defaults), so this uses the **blit path**: a persistent `Uint32Array` view over an `ImageData`, plot/erase pixels, `putImageData` once per frame — not per-point `fillRect`.

## Deviations from the C
- **Full-buffer clear each frame instead of dirty-bbox erase.** The C tracks the bounding box of the points drawn and erases only that rectangle on the next frame (plus double-buffering / `XCopyArea`). Here the whole `Uint32` buffer is `fill(BLACK)` each step before redrawing — for a typed-array blit that is just as cheap and removes all the bbox/`XCopyArea` bookkeeping. Visually identical (the figure is fully redrawn every frame in both).
- **STEP products in float64; shift replaced by `Math.floor(v/1024)`.** The coefficients are int (as in the C), but the per-step sum `ua*x + ub*y + utx` is evaluated in float64, not 32-bit int. On a 4K canvas that sum can reach ~3e9, which **overflows int32** (2.15e9) in the C — undefined behaviour producing a few stray wrapped pixels. float64 holds the sum exactly and `Math.floor(.../1024)` reproduces the arithmetic `>>10` (floor division, correct for both signs), so the result is **bit-identical to the C wherever the C does not overflow**, and cleaner where it would. (Note: the RNG differs from glibc anyway, so per-pixel sequences are not bit-identical; the chaos game converges to the same attractor either way.)
- **Point budget capped.** Points-per-pass is clamped to 4,000,000 so a high "Detail" on a slow machine can't lock the tab (the C trusts the user; "Detail" is exponential — 14 means functions^13 points). At functions=3 the cap is never reached even at detail 14; it only bites at functions=6, detail ≥ 10.
- **`mode`, `recurse`, `multi` not surfaced.** The stock UI (ifs.xml) exposes only delay / functions / detail / colors and the translate/scale/rotate toggles. We follow the C defaults `recurse=False` (so we port the `iterate` chaos-game path, not the command-line-only `recurse` tree) and `multi=True` (the per-lens coloured loop). `mode` is read but **unused** in the C.
- **Frame-rate / morph pace (`OVERHEAD = 6882`).** The morph is frame-coupled (counters advance per `step()`), so the frame rate sets the morph speed. The `delay` default is now the **stock 20000 µs** (was a by-eye 30000); the stock delay is only a sleep floor, so the rAF loop adds `OVERHEAD` to reproduce the live binary's effective rate: the `-fps` overlay measures **37.2 fps**, while the port at the bare stock delay ran 50 steps/sec (1.34× fast), so `20000 + 6882 = 26882 µs → 37.2 steps/sec`. A calibration, not a tuning knob; the slider still maps 1:1 to the xml. See [[framerate-calibration]].
- **Encoding:** the only DOM-bound non-ASCII is the micro sign in the frame-rate unit, written in the JS as a `µ` escape (renders as `µs`). Literal Unicode (em dashes, µ) appears only in comments and in this file.

## Config
Units/defaults/ranges mirror `hacks/config/ifs.xml`: `delay` (µs/frame, `invert` "Frame rate" slider; stock 20000, with a loop `OVERHEAD` — see above), `functions` (2–6, 3), `detail` (4–14, 9, exponential), `colors` (2–255, 200), and the `translate` / `scale` / `rotate` morph toggles (all on). The translate/scale/rotate toggles are **live** — the morph reads them each frame, so disabling one freezes that axis instantly (matching the original, where they gate the `mutate` branches). `functions` / `detail` / `colors` are **not live** (they resize the lens set, point budget and palette), so changing them re-seeds via `reinit()` with a fresh figure. The stock `showfps` checkbox is a framework control, not a hack resource, so it is omitted.

## Audit (2026-06-27, Batch 1C — nonlinear dynamical systems)
Re-read against `ifs.c` line-for-line. Fixes applied:
- **Palette (HIGH).** Replaced an invented vivid `hsl(h,100%,55%)` rainbow with `make_smooth_colormap` (colormap.js) — the routine ifs.c actually calls. The original look is muted/pastel and slowly cycling, not a saturated spectrum.
- **Multi-mode overlap colour (HIGH).** The dedupe board is now reset **before each lens pass** (was once per frame). The C `memset`s it per pass, so overlaps are last-pass-wins and every lens draws its complete cloud; the old per-frame reset let lens 0 block the later passes (first-pass-wins, sparser later clouds).
- **Chaos-game point (LOW).** Now persists across passes/frames (the C's `st->x/st->y`) instead of being reset to the centre each frame. Invisible thanks to the 10-step burn-in, but faithful — the centre seed only ever applied to the C's command-line `recurse` path, which we don't port.
- **Coefficients (LOW).** `ua..uty` truncated to int on assignment to match the C's `int` fields (STEP products stay float64 to dodge the int32 overflow UB).
- **Labels.** "Detail (exponential)" → "Detail", "Colors" → "Number of colors" to mirror the xml.

Needs a live check (this agent does not run the binary): the multi-mode last-pass-wins overlap colouring, and the morph/delay pace (see Deviations).
