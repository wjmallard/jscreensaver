# xflame — port notes

Port of `xflame.c` by Carsten Haitzler (1996; TrueColor/utility/image work by Rahul Jain, Daniel Zahn and jwz, 1996-2018) — a classic cellular "licking fire" effect. By default a picture (the built-in "Bob" face) appears to burn as flames rise from the bottom of the screen.

Original: <https://www.jwz.org/xscreensaver/> · source: `xscreensaver-6.15/hacks/xflame.c` (~826 lines) · config: `hacks/config/xflame.xml`. Rendering style mirrors `See [[eruption]]` (Uint32 fire-palette blit) and `See [[metaballs]]` (offscreen field + `drawImage` upscale).

## Algorithm
A heat field lives at HALF the image resolution (`fwidth x fheight`, 1-cell padded). Each frame (`xflame_draw`):
1. **FlameActive** — the bottom "active" row (one buffer row below the visible area) is re-seeded: each cell drifts by `random()%variance - vartrend` and is stored mod 255. Optional `bloom` occasionally surges `residual`/`hspread`/`vspread`, which then ease back 10%/frame toward their base values.
2. **FlamePasteData** — the built-in fire SOURCE image ("Bob", `bob.png`) is pasted into the field each frame: for every source pixel with value >= 24 it adds `random()%(val/24)` heat to the matching field cell, so the picture's hot (dark-in-the-image, after inversion) regions keep burning. This is the C's DEFAULT (`*bitmap: (default)`), so the default look is "a picture on fire," not a plain flame.
3. **FlameAdvance** — heat propagates UPWARD. The field is swept bottom-to-top; every lit cell pushes `vspread/256` of its value into the cell directly above and `hspread/256` into the two diagonal-up cells (clamped at 255), then keeps `residual/256` of itself. Because the sweep goes bottom-up, a hot seed cascades up many rows in a single pass. `top` tracks the highest non-empty row as a work bound.
4. **Flame2Image** — the half-res field is 2x-upscaled into the image (each cell -> a 2x2 block with bilinear-ish interpolation of its right/below/below-right neighbours), and each value indexes a `ctab` fire LUT (black -> red -> orange -> yellow -> white).

The stock constants are deliberately tuned for **marginal stability**: `vspread + 2*hspread + residual == 256` (97 + 60 + 99). Total heat per row is conserved as it rises, so the fire neither dies out nor saturates the screen — it licks tall and the random seed makes the tips flicker. A `bloom` surge briefly pushes the sum past 256, making the flame flare up, then it relaxes back to balance.

## Palette
The fire LUT is **not** a `make_smooth_colormap`/`make_color_ramp` palette — `colormap.js` does not apply. `InitColors` builds `ctab[j]` analytically from the `foreground` colour (default `#FFAF5F`): per channel `(2j - (255-fg_channel))*3`, clamped 0..255, giving the black -> red -> orange -> yellow -> white ramp. The port bakes in the default foreground (the xml exposes no `-fg`/colour picker) and builds the LUT once at init; it is never cycled. Faithful, exact transcription.

## Rendering / retina
The field is computed at **half the LOGICAL resolution** (`round(innerWidth)>>1`, even dims as the C forces), written into an offscreen Uint32 `ImageData`, then `ctx.drawImage`-upscaled to the device-res canvas. **This is deliberate, not just a dpr-cost dodge:** xflame's flame DYNAMICS — bloom amplification, flame height, startup ramp — are tuned to a specific cell size. A brief experiment computing the field at *device* resolution (to look crisper on retina) put ~2x more rows in each column, and since the bloom multiplies heat on every row it rises through, that over-amplified it into a **white-hot flare** the live binary never shows. So the field must run at the same logical cell size the live binary uses; the CSS-res image upscaled to the device canvas then carries the same softness the live binary has when the OS scales its window to a hidpi display. `MAX_CELLS = 540000` shrinks the field on very large windows then upscales; `bob` is sized relative to the field image (`imgW/imgH`), matching the C's compare against `st->width`.

## Deviations from the C
- **Fire source restored (STRUCTURAL ADDITION — live-verified vs the binary 2026-06-27).** The previous port DROPPED the bitmap feature entirely, which made its default look a *plain* flame — but `*bitmap: (default)` means the C ALWAYS pastes the built-in "Bob" image by default, so the previous default did not match the C's default (nor the demo video, youtube `jUJiULU4i0k`). Restored faithfully: `bob.png` is copied verbatim into `hacks/images/bob.png` (byte-identical to `xscreensaver-6.15/hacks/images/bob.png`, 64x64, 1491 bytes) and loaded relative to the module via `new URL('./images/bob.png', import.meta.url)` — the same asset idiom as `xanalogtv.js`; `loadBitmap` + `double_ximage` (nearest-neighbour pixel-doubling until the image is ~1/10 of the field) + `gaussian_blur` + the grayscale/contrast/inversion conversion + `FlamePasteData` are transcribed from the .c. **One platform deviation:** the C's vertical `XGetPixel(x, image->height-1-y)` flip is OMITTED — the C's X image is bottom-up so that flip corrects it, but canvas `getImageData` is already row-0-top, so keeping the flip rendered Bob upside down (the user caught it once the blur fix made him legible). Reading top-down makes the port's Bob match the live binary's upright orientation. Confirmed against the live binary at matched resolution: Bob burns upright in the flames, matching the original. User-supplied `--bitmap` files and the `--bitmap none` mode are not exposed (no file picker); only the built-in default is used.
- **Bob's size is window-scaled (faithful, reviewed 2026-06-27 — do not "fix").** The C doubles Bob in powers of two until `>= window/10`, so his size scales with the window and lurches in 2x steps at the doubling boundaries: on a small window he looms relatively large, on a wide one he is proportionally smaller (the flames are a fixed cell-height and don't scale, which exaggerates the effect). This is the C's own (acknowledged-quirky) behavior; the port reproduces it rather than capping Bob to a fixed size. The user reviewed this and chose to keep it faithful.
- **Fixed fire LUT (default foreground).** The C derives `ctab` from the `foreground` resource (default `#FFAF5F`); the xml exposes no colour control, so the port bakes in that default. The flame colours match stock xflame. (See Palette.)
- **Full-frame blit.** The C's `DisplayImage` only blits the sub-rectangle from `top` downward (a perf optimization that relies on rows above `top` already being black). We render every visible row each frame; since rows above `top` are always 0 -> `ctab[0]` (black), the output is identical, just simpler and stale-row-proof. `top` is still maintained as the FlameAdvance work bound (faithful).
- **No XOR / feedback tricks needed** — the C uses a plain XImage, directly emulated by the Uint32 ImageData.
- `showfps` (xml boolean) is host chrome, not ported.
- Loop uses the shared fixed-timestep rAF lag-accumulator instead of `usleep(delay)`; default `delay` 10000 µs (the xml default). **Framerate calibration (`OVERHEAD = 9000`, ESTIMATED — FLAGGED for a visual check):** the stock delay is a sleep floor and the live rate is lower (delay + framework overhead), but the live `-fps` overlay was **unmeasurable** here (the full-frame fire obscured the overlay), so `OVERHEAD` is an estimate (~9000 µs) by analogy to the other dense per-pixel/ImageData hacks (`10000 + 9000 = 19000 µs → ~52.6 fps`). Needs a side-by-side visual check against the live binary to confirm the pace. See [[framerate-calibration]].

### Removed in this audit (were non-faithful)
- **Stability renormalization (REMOVED).** The previous port renormalized `vspread/hspread/residual` down to a 256 "heat budget" whenever they exceeded it. The C does NO such thing, and the renormalization actively *suppressed the bloom flare* (a bloom deliberately pushes the sum past 256 to make the fire surge). It only existed to keep the (now removed) extra sliders from running the field away to white. Removed; the eased values relax back to the stock 97/30/99 on their own, so the field stays a live flame with no fudge.

## Config
Mirrors `hacks/config/xflame.xml` 1:1 — the only resources the original surfaces are `delay` and `bloom`:
- `delay` — µs between frames (xml "Frame rate", `low=0 high=100000 default=10000`, inverted), **live**.
- `bloom` — enable random blooming surges (xml "Enable blooming", default True), **live**.

xflame.c ALSO has command-line-only resources (`-hspread 30`, `-vspread 97`, `-residual 99`, `-variance 50`, `-vartrend 20`, `-baseline 20`) that are NOT in the xml. Following the house convention (config maps 1:1 to the xml, like `swirl.js`), these are fixed at the stock defaults (the `I*`/`VARIANCE`/`VARTREND`/`BASELINE` constants) rather than exposed as sliders. The previously-exposed Flame-height/spread/persistence/turbulence/cooling sliders were removed (they are not GUI resources, and exposing them forced the non-faithful renormalization above).

## Motion
The C continuously redraws an animated fire (each `xflame_draw` returns `delay` and is called again). The port continuously steps via the rAF accumulator at the same nominal pace. This is genuinely animated — not a static hack — so continuous animation matches the C.

## Correctness self-review
- **Won't go all-black or all-white.** A headless numeric harness ran `FlameActive`+`FlameAdvance` (bloom on) for 800 steps on a 200x120 field: heat stayed in 0..255 with ~68% of cells nonzero and the lit region oscillating between ~1/3 height and full height (flicker + bloom flares), and the eased `vspread/hspread/residual` relaxed back to exactly 97/30/99 after each bloom — confirming the field stays a live, bounded flame without the removed renormalization.
- **No overflow.** `(v1 * spread) >> 8` operates on products `<= 255 * ~120` (the eased spreads stay near their bases plus small bloom bumps) — far under 2^31. Heat is `Uint8Array`, so spread writes clamp at 255 (matching C's `MAX_VAL`) and self-decay / paste-add truncate mod 256 exactly like the C's `unsigned char`.
- **Seed wrap is exact.** `flame[i] = v1 % 255` reproduces the C's `(unsigned char)(v1 % 255)`: JS `%` truncates toward zero like C, and the `Uint8Array` store reproduces the unsigned-char wrap for negative results (e.g. -5 -> 251).
- **Fire-source asset verified.** `hacks/images/bob.png` is byte-identical to the source (1491 bytes, IHDR 64x64). It loads asynchronously from the served file; until decoded, `theim` is null and the flame simply runs without the source (graceful), then the paste begins once `onload` fires and `buildTheim` fits it to the field.
- **pause/resume/reinit/resize** are the standard skeleton; `resume` resets `lastTime` so there is no catch-up burst, and `reinit`/`resize` rebuild the field, offscreen, and refit the fire source cleanly.

## Residual risk
The fire-source pipeline (asset decode -> nearest-neighbour doubling -> grayscale/contrast/inversion -> gaussian blur -> per-frame paste) was **live-verified** against the binary (2026-06-27): Bob appears upright (after the top-down flip fix above), at the right scale/position, with the dark facial features reading as the hot/burning regions (the invert + contrast). The bloom was also verified across many frames at hidpi: it now produces a mild orange surge, not the full white-out an earlier device-resolution field caused (see Rendering). Field counts/sizes derive from logical px, so there is no dpr-count bug.
