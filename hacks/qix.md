# qix — port notes

Port of `qix.c` by Jamie Zawinski (1992) — bounces a few polygonal "lines" around the screen, trailing the classic Qix ribbons that overlap and mix.

Original: <https://www.jwz.org/xscreensaver/> · source: `qix.c` (~641 lines) · <https://en.wikipedia.org/wiki/Qix>

## Algorithm
A *qix* is a ring buffer of `segments` polygon **frames**; each frame is a snapshot of `poly` vertices (default 2 = a plain line segment), every vertex carrying a position and a velocity. Frame 0 is seeded randomly and all the others start as copies of it, so the trail begins collapsed and unfurls. Each step (the C's `qix1`):

1. **Erase** the polygon in the slot about to be overwritten (the oldest) by repainting it in black.
2. **Build** the new polygon by copying the previous frame's vertices, then advancing every vertex by its velocity and **reflecting** it off the walls (the `wiggle` macro: optionally jitter the velocity, clamp it to `±spread`, step, and on a wall hit pin to the wall and reverse — `point += 2·|delta|`).
3. **Hue-shift** by `colorShift` degrees and **draw** the new polygon, then advance the write pointer.

At steady state exactly `segments` polygons are live: one drawn and one erased per step, a constant-length ribbon. With `solid` (the default, `poly` forced to 2) consecutive frames are joined into a **filled quad** — the solid ribbon; `hollow` strokes the closed polygon outline. `gravity` adds a downward pull to every vertex's `dy`; `random` motion jitters velocities each step (vs. clean `linear` bounces). `count` runs several independent qixes at once (stock default **4**).

### Two colour models
qix.c ships **two** ways of colouring the ribbons, selected by `transparent` (stock default `true`):

- **Transparent (the default).** Each qix is **one fixed colour** and the `count` ribbons **overlap and mix**. On the live X11/XQuartz binary's TrueColor visual this is the plane-mask path: `allocate_alpha_colors` (`utils/alpha.c`, the `!cmap` branch) gives each qix a `plane_mask = random() & (red|green|blue bits)` — i.e. a uniformly random 24-bit RGB value — and builds one GC per qix with that plane mask and `foreground = ~0`. Because `xor` is off by default these are plain `GCForeground|GCPlaneMask` (GXcopy restricted to the qix's bits), so **drawing ORs the qix's colour bits into the framebuffer and erasing ANDs them out**. Overlaps are therefore a **per-channel bitwise OR** (two ribbons crossing make a brighter blend; e.g. live `0x3047CD` over `0xE88CC2` gives `0xF8CFCF`, matching OR exactly and *not* clamped-add `0xFFD3FF`), and an erasing ribbon **reveals whatever qix lies under it** rather than punching black. On a PseudoColor visual the same routine instead allocates real colormap cells and does a clamped additive/subtractive merge (the `additive`/`subtractive` option) — that path is dead on TrueColor (the source even concedes `-subtractive` can't be done there), so the live default is always the OR look.
- **Hue-cycle (`transparent` off).** Each qix is one **opaque** colour seeded `hsv_to_rgb(random()%360, frand(1.0), frand(0.5)+0.5)` whose **hue** advances by `colorShift` each frame (saturation/value held). This is the C's non-transparent path.

## Module shape
`start(canvas) -> { stop, pause, resume, reinit, config, params }` — see `squiral.md`. Closest twins: `spiral.js` (the same ring-buffer **draw-newest / erase-oldest** trail) and `braid.js` (vector strokes via `Path2D`/`beginPath`).

## Rendering — draw-newest / erase-oldest, `O(1)` shapes per step
Both colour models share the C's incremental scheme: each step **draws the single new polygon and erases the single outgoing one**, leaving the rest of the trail untouched (`O(1)` shapes/step, not `O(segments)`), exactly like the C and like `spiral.js`.

- **Hue-cycle mode** uses canvas **vector ops**: one `fill()` of a 4-point quad (solid) or one closed `stroke()` polyline (hollow) per step per qix; the erase repaints the outgoing shape in black.
- **Transparent mode** can't use canvas compositing — Canvas 2D has no bitwise OR (`'lighter'` is clamped-*add*, proven wrong above; `'xor'`/`'difference'` are alpha/`|a-b|`). So the port keeps a **software RGBA framebuffer** (`ctx.createImageData(W,H)`) holding the X11 pixel bits directly and rasterises into it: an even-odd **scanline polygon fill** for solid quads (and a thick-Bresenham line for hollow), doing `buf |= colour` to **draw** and `buf &= ~colour` to **erase**, then `putImageData` of just the changed rect. This is the quasicrystal/crystal-class software-raster approach — exact bitwise mixing, no blend-mode fake. One shared buffer reproduces the live's exact semantics for free: where two qixes' random masks share bits, an erase of one clears the shared bits of the other (the faithful X11 plane-mask interference), and adjacent ribs leave a 1-px trailing-edge seam where the AND-NOT erase clips the still-live neighbour — **both artifacts are present in the live binary too** (visible as the same faint stipple along certain rib boundaries in side-by-side captures).

A faithful quirk carried over from the C: a slot is **drawn** paired with its *predecessor* frame (`ofp = fp-1`) but later **erased** paired with its *successor* (`fp+1`). Tracing the ring shows every drawn quad is still erased exactly once, one slot-cycle later, with the *identical* four coordinates, so the erase clears precisely what the draw set and nothing accumulates — see *Correctness* below.

## Deviations from the C
- **Transparent is the faithful default; `xor` and PseudoColor `additive`/`subtractive` are omitted.** The transparent (plane-mask OR) default is now ported via the software framebuffer above, matching the live binary. Not modelled: `-xor` (which on TrueColor with `-non-transparent` just forces `mono_p`, i.e. a plain white ribbon, and with `-transparent` swaps the OR for a GXxor of plane bits — a non-default look), and the `-additive`/`-subtractive` clamped merge (only reachable on a **PseudoColor** writable-colormap visual, which XQuartz/modern displays never present — the source itself notes `-subtractive` "can't be done" on TrueColor). So `xor`/`additive`/`subtractive` stay out of the config box; `transparent` is now exposed as the key toggle (on by default).
- **Float device-pixel coords instead of fixed-point `<<6`.** The C works in `<<SCALE` (SCALE=6) fixed-point and `>>SCALE`s only at draw time. The port keeps positions/velocities directly in device pixels (scaled by `devicePixelRatio`), so `spread`/`size` are multiplied by `S` rather than shifted. The reflection arithmetic and clamps are converted cleanly, not half-converted.
- **`devicePixelRatio`**: the backing store is sized in device px; `spread` (velocity clamp), `size` (extent clamp), the random-jitter amount, and the gravity pull are all multiplied by `S`, so motion and density look the same on retina.
- **Gravity strength bumped.** The C adds `3` to `dy` in `<<6` units (≈0.05 px/step) every frame, which is nearly imperceptible; the port adds `0.5·S` px/step so the Gravity toggle visibly arcs the ribbon.
- **`poly` cap 24** (the C's `MAXPOLY` is 16) and the xml's `high="100"` is trimmed to a saner slider range; **`count` default 4** (stock, matches the xml + the live transparent look) and max 12 (12 OR-ribbons already wash toward white). `delay` keeps the stock value (see **Timing**).
- **Colour** — transparent mode uses random RGB plane-mask colours (the live's `random() & masks`); hue-cycle mode is a faithful `hsv_to_rgb` port, not `hsl` — see **Palette**. `colorShift` (Color contrast) maps 1:1 as degrees-per-frame (hue-cycle only).
- Keypress / `fps` handling and the periodic `get_geom` re-query are dropped — the host owns keys and the meter, and `resize()` already re-seeds on a window change.

## Palette
qix is a **native screenhack** that colours each qix itself — there is no `make_*_colormap`.

**Transparent (default).** Each qix's colour is `random() & masks` = a uniformly random 24-bit RGB value (alpha.c:178), held for the qix's life; there is **no hue cycle**. The port draws three random bytes per qix and uses that triple as both the colour and the OR/AND-NOT bit-mask. So the screen is a few fixed-colour ribbons whose crossings are the per-channel **bitwise OR** of the colours involved (brighter, washing toward white in dense triple-overlaps) on black — verified pixel-exact against the live binary (`0x3047CD | 0xE88CC2 → 0xF8CFCF` etc.). This was the bug the user flagged: the port previously always ran the **non-default** hue-cycle, so the default look was a full-spectrum rainbow instead of these mixing fixed-colour ribbons. Now fixed — transparent is the default.

**Hue-cycle (`transparent` off).** `init_one_qix` seeds every qix with `hsv_to_rgb(random()%360, frand(1.0), frand(0.5)+0.5)`: a random hue, a random **saturation in [0, 1)** and a random **value in [0.5, 1)**, held for the qix's lifetime; `add_qline` advances *only the hue* by `colorShift` each frame (`h = (h + colorShift) % 360`), holding s and v. Because the saturation is a free `frand(1.0)` roll, the screen mixes vivid, pastel and **near-grey** ribbons (a low-S roll = a grey ribbon). The port transcribes `utils/hsv.c`'s `hsv_to_rgb` directly (inline `hsvColor(h, s, v)`, with the X server's 16-bit→8-bit `>>8` downsample), held `sat`/`val`, cycling only the hue. (An even-earlier port used a fixed vivid `hsl(h, 100%, 55%)` — wrong colour *space* **and** no per-qix S/V variety; that vivid-rainbow bug was fixed in Batch 1G, and this hue-cycle path is what remains as the non-default alt.)

## Timing
Stock `*delay: 10000`. Measured on the live binary's `-fps` overlay, both colour models are **delay-bound** (Load ~50 %, well under 100 %): the **transparent default reads 50.5 fps / 49.5 % Load** (sleep slice 10000 µs = the stock delay), the non-transparent path ~54 fps. Since transparent is the default, `OVERHEAD = round(1e6 / 50.5) − 10000 = 9802 µs`, added to `config.delay` in the rAF lag-accumulator so a step runs every ≈19.8 ms instead of racing the bare 10 ms floor (~1.8× too fast); `config.delay` stays at the stock 10000 and the slider maps 1:1. (The non-transparent hue-cycle alt then runs a hair slow at this value — negligible.) The software rasteriser keeps up with no catch-up. (An earlier note here guessed the transparent default ran "~25 fps" — that was unmeasured and wrong; it is 50.5 fps.)

## Config
Ranges mirror `hacks/config/qix.xml`: `delay` (Frame rate, live, inverted µs), `spread` (Density, live, inverted — bigger spread = faster/denser ribbon), `colorShift` (Color contrast, live, hue-cycle only), `gravity` (live); `segments` (trail length), `count` (default 4), `size` (Max size), `poly` (Poly corners), the `fill` (Solid/Line) select, and the **`transparent` checkbox (default on)** are **non-live** (they size the queue / vertex count / qix count or switch the colour model + framebuffer, so a change re-runs `init()` via `reinit()`, clearing the canvas). `motion` (Linear/Random) is live. "Reset to defaults" and `r` (restart) re-seed via `reinit()`. Omitted vs. the xml: `xor` and the additive/subtractive colour-mode select (see **Deviations**).

## Correctness self-review
- **Bounded queue, no growth.** The ring buffer is allocated once at `init()` to `segments` frames; `step()` only overwrites the slot at `fp` and advances `fp` modulo `nlines`. Trail length is constant.
- **No residue.** Each physical rib (the quad between adjacent slots *k* and *k-1*) is drawn when slot *k* is written and erased when slot *k-1* is recycled, one slot-cycle later — so every drawn shape is erased exactly once. Verified by tracing the ring with `nlines = 4` by hand (including the first cycle).
- **Clean first frame, no degenerate start.** The seed frame is randomised inside `[0,W]×[0,H]` (or within `size` for `poly=2`), and all frames start `dead=true`, so the first `segments` steps only *draw* (the `if (!old.dead)` guard skips the not-yet-drawn erase) — the trail unfurls from a single polygon, on-screen from frame one.
- **Bounce from frame one.** `wiggle` pins any out-of-bounds vertex back onto the wall and reverses its velocity in the same step, so even an initially fast vertex can never escape `[0, max]`.
- **Pause/resume & reinit.** Pause cancels rAF and parks `rafId = 0`; resume resets `lastTime = 0` to avoid a catch-up burst (standard skeleton). `reinit()` clears to black and rebuilds the queue, giving a fresh screen.
- **Solid-quad guard.** Solid draws skip when the partner frame is `dead`, and the trace above shows the partner (`fp+1`, the oldest) is always alive by the time an alive frame is erased — so no malformed quad is ever drawn or erased.
- **Transparent erase is self-cancelling.** The framebuffer erase ANDs out *exactly* the bits the draw ORed in, over the *identical* four coordinates (the draw/erase quads are the same polygon, traced above), using the *same* scanline routine — so a qix never accumulates stale bits in isolation. Cross-qix bit interference and the 1-px trailing seam are intentional (they match the live X11 plane-mask GCs). The buffer is `createImageData(W,H)` once per `init()` and only the changed rect is blitted each step.
- **Mode isolation.** `transparent` is resolved once in `init()`; the buffer (`img`/`buf`) is allocated only in transparent mode and is the *sole* writer to the canvas there (always `putImageData`, never `ctx.fill`), while hue-cycle mode never touches the buffer — so the two paths can't interfere, and toggling `transparent` re-runs `init()` (fresh buffer or black canvas).
