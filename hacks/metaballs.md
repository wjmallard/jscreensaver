# metaballs — port notes

Port of `metaballs.c` by W.P. van Paassen (2002-2003) — several "balls" drift around the screen, each stamping a soft radial blob into a density field; where blobs overlap their densities sum, so neighbours bulge toward one another and merge into one gooey shape. The classic lava-lamp metaball surface.

Original: <https://www.jwz.org/xscreensaver/> · source: `xscreensaver-6.15/hacks/metaballs.c` (~438 lines).

## Algorithm
- **One radial stamp, built once.** A `dradius x dradius` byte bitmap (`dradius = 2 * radius`) where each cell's density is `(1 - frac^2)^4 * 255`, `frac = dist^2 / radius^2` — 255 at the centre, smoothly to 0 at the rim, 0 outside the circle.
- **Per frame:** clear the per-pixel density grid (`blub`); random-walk every ball by `~[-delta, +delta]` px; for each ball still roughly on-screen, add its stamp into the grid (a **saturating** add clamped to `ncolors-1`, so overlaps brighten but never wrap); a ball that wanders off-screen respawns near centre.
- **Colour.** The accumulated density (0..`ncolors-1`) indexes a palette that ramps **black -> base colour -> white** (index 0 = black, middle = the base colour, top = white). So dense cores read bright, soft fringes fade to black, and the black background (density 0) stays unlit.
- **Duration.** Every `cycles` frames the palette base colour is re-rolled and all balls respawn (the C's `metaballs_draw` reset), so the scene periodically recolours and rearranges.

Rendering uses the **blit path** (`hacks/hack-rendering-perf` rule): the whole field is rebuilt every frame, so it writes a `Uint32Array` view over one `ImageData` and blits once — not millions of per-pixel canvas calls. See [[binaryring]] and [[thornbird]] for the same Uint32-ImageData idiom; skeleton/loop/conventions follow [[squiral]] and [[greynetic]].

## Config
Units and defaults mirror `hacks/config/metaballs.xml` 1:1: `delay` (µs/frame, 10000), `cycles` ("Duration", frames before re-roll, 1000), `ncolors` (palette size, 256), `count` (Ball count, 10), `radius` (Ball Radius %, 100), `delta` (Ball Movement px, 3).
- **`live: true`** — `delay`, `cycles`, `delta`: read every step, apply instantly (they only affect pacing / when to re-roll / the per-step walk).
- **`live: false`** — `count`, `radius`, `ncolors`: these size the ball array, the stamp bitmap, and the palette, so a change re-runs `init()` via `reinit()` (which also clears the canvas).
- `delay` uses `invert: true` (the xml's `convert="invert"` "Frame rate" slider — drag right = faster), shown as the raw µs value with a `µs` unit string.

## Deviations from the C
- **Reduced internal resolution (the one real deviation).** The C computes the field at full window resolution. Here the field is computed at **logical (CSS-pixel) resolution** — `canvas.width / devicePixelRatio` — into an offscreen canvas, which the main canvas then upscales (`drawImage`). On a retina/4K display this is ~4x less per-frame pixel work (~4.8M vs ~17M ops/frame at 1080p logical) and the bilinear upscale actually *softens* the blobs, which suits the gooey look. The radius is still taken as a percent of `gridHeight / 8` exactly as the C takes it of the window height, so balls keep the same on-screen size. **If full-res crispness is ever wanted, set the field grid to `canvas.width/height` and drop the upscale** — the rest is unchanged.
- **Default delay** is the stock 10000 µs (the slider maps 1:1 to the xml resource). The rAF loop adds `OVERHEAD = 9231 µs` so `(delay + OVERHEAD)` reproduces the live binary's effective rate, not the nominal one: measured against the live `-fps` overlay metaballs runs **52.0 fps**, while the port at the stock delay ran ~100 fps (1.9× fast); `10000 + 9231 = 19231 µs → 52 fps`. A calibration, not a tuning knob — see the framerate-calibration note.
- **Palette base colour** is picked faithfully as three independent random bytes (`[randInt(256), randInt(256), randInt(256)]`), matching the C's `SetPalette` base of three independent `random() % 0xFFFF` channels (here at 8-bit-per-channel resolution) rather than parsing an X11 colour-name resource; the black->base->white ramp itself is the C's `SetPalette` math verbatim. Because the channels are independent and uniform, the base is frequently muted/dark/pastel, not only vivid. (Earlier this port used a fully-saturated `hsl(random, 100%, 50%)` base, which only ever produced vivid hues - the one colour deviation, since uniform-RGB random != max-saturation HSL.)
- **No XOR / feedback / erase-transition tricks** are involved, so nothing else needed emulation. The C's `if (aiColorVals[..] > 0) XPutPixel(..)` "skip density-0 pixels" is reproduced simply by mapping density 0 to opaque black (`palette[0] = 0xFF000000`) and writing every pixel — identical result on a black background.
- Random-walk integer truncation matches the C precisely: the C truncates only the always-non-negative product `(delta+0.5)*frand(2.0)` before subtracting the integer `delta`, so this floors the product (`-delta + Math.floor(product)`), not the signed sum (which `| 0` would round toward zero and diverge by 1 px on negative fractions).

## Correctness self-review
- **Never all-on / all-off.** Each blob is a bounded radial stamp peaking at 255 and decaying to 0; the background (density 0) maps to black. A headless 5-frame simulation at 1920x1080 with the defaults gave ~88% black, with a graded spread through the mid-tones up to ~3.3% saturated cores — i.e. bright connected cores fading to black fringes, exactly the metaball surface. The "threshold" here is the palette ramp itself (low density -> near-black), not a hard binary cut, matching the C.
- **No freeze / runaway.** The rAF loop is the standard lag-accumulator with `MAX_CATCHUP_STEPS` cap; `step()` does a fixed amount of work (clear + `count` stamps + one full-grid palette map) with no data-dependent unbounded loop. Balls that drift off-screen are caught by the same bounds test the C uses and respawned, so they can't wander away forever and stall the field.
- **Saturating add is guarded.** The grid is `Uint8Array` and the add is clamped to `ncolors-1` (<= 255) before storing, so it can't overflow the byte or wrap.
- **Pause/resume** uses the shared `rafId === 0` sentinel and resets `lastTime = 0` on resume, so resuming doesn't fire a catch-up burst; **reinit** rebuilds stamp/palette/grid/balls and clears the canvas, giving a clean fresh screen for non-live config changes. Verified the grid index math (`gy*gw + gx`, stamp `i*dradius + j`) and bounds clipping by hand and by the simulation (no out-of-range writes).

**Spot-check in the browser:** confirm blobs visibly merge/split and drift smoothly from frame one; nudge **Ball Radius** and **Ball count** (both reinit) and **Ball movement** (live) and watch the field respond; let it run past ~1000 frames to see the **Duration** re-roll recolour the scene; check `p` pause/resume doesn't jump and `r`/config-reset gives a clean restart.
