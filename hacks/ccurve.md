# ccurve — port notes

Port of `ccurve.c` (Rick Campbell, 1998-1999) — draws self-similar linear fractals, the Levy / "C Curve" family, by recursively subdividing a line into a rotated/scaled copy of a small random "generator" of 2-4 segments, in a vivid rainbow gradient. Each fractal grows one recursion level per frame, auto-fitting the view as it gains detail; at full depth it lingers, then a fresh generator starts.

Original: <https://www.jwz.org/xscreensaver/> · source: `xscreensaver-6.15/hacks/ccurve.c` (~871 lines) · <https://en.wikipedia.org/wiki/Levy_C_curve>

## Module shape
`start(canvas) -> { stop, pause, resume, reinit, config, params }` — see `squiral.md`. Closest technique twin for the loop is `xspirograph` (variable-delay step) and for the bucketed vector rendering is `braid`.

## Algorithm
A **generator** is a chain of 2, 3, or 4 `Segment{angle, length}` picked by `select_2/3/4_pattern` (transcribed verbatim — they hold the hand-tuned C-Curve, Koch-ish, and dragon-ish presets plus randomized variants). `normalized_plot` turns the chain into cumulative points, then rotates+scales the whole thing to run from `(0,0)` to `(1,0)`. The curve at depth N is then built by `self_similar_normalized`: at depth 0 it emits the leaf line `(x1,y1)->(x2,y2)`; otherwise it `realign`s the normalized generator onto that segment and recurses on each sub-segment. A leaf's colour is `floor(line_count * 256 / total_lines)` (`color_count` = `MAXIMUM_COLOR_COUNT`) — a rainbow gradient walking along the curve.

The driver (`ccurve_draw`) is a **progressive build**: `draw_index` starts at 0 and rises one level per frame, so you watch the curve sharpen from a single line (depth 0) to the bare generator (depth 1) to ever-finer self-similar copies. `draw_iterations = floor(log(limit) / log(segment_count))` caps the depth (`limit` = the "Density" budget); `total_lines = segment_count^draw_index`. After each frame the plotted extents (`plot_min/max`) are measured and the view box (`draw_min/max`) is re-fit to them (+20% margin, aspect-corrected) so the next, deeper frame stays framed. When `draw_index` reaches `draw_iterations` the screen lingers (`delay`), the generator is dropped, `draw_index` resets to 0, and a new generator is chosen — usually after jittering the four base endpoints by up to +/-0.45 (the C's `(random()%3)!=0` branch) so successive fractals drift around the plane.

## Rendering — vector ops, bucketed by colour
Genuinely line-shaped (one `XDrawLine` per leaf in the C), so this uses **canvas vector ops**. At full depth a 4-segment generator emits 4^7 = 16384 leaves (3-segment: 3^10 = 59049) — far too many `stroke()` calls. Since the C casts each leaf's colour to an integer index, leaf segments are **bucketed by that index into one `Path2D` per colour** (256, the C's `MAXIMUM_COLOR_COUNT`) and each bucket is stroked once, the same trick `braid` uses. Verified across 40 sessions / 65M emitted coords.

## Deviations from the C
- **Two timers, in seconds.** ccurve is special-cased: `delay`/`pause` are read as *floats* and multiplied by 1e6, so they are **seconds**, not the usual microseconds (the xml labels them in seconds, 0-30 s / 0-5 s). The port keeps that and uses the **variable-delay loop** (boxfit / xspirograph style): `step()` returns the ms to wait — `pause` (short, between depth frames) then the longer `delay` (linger once a fractal completes). Both defaults are stock (0.4 s / 3 s). The unit string stays ` s` (NOT `µs`) precisely because of this seconds special-case. The xml's two sliders are surfaced with their exact `.xml` labels: **Animation speed** (`pause`, `convert=invert`) and **Change image every** (`delay`).
- **Clears every frame.** The C draws into an off-screen pixmap and `XCopyArea`s it; canvas is double-buffered, so each depth frame clears to black and redraws the whole curve at the new depth — identical result, flicker-free. (The C also draws each frame from scratch into the pixmap, so this matches its logic, not just its look.)
- **Degenerate-extent guard (the one real safety deviation).** If a curve has zero width or height (e.g. a perfectly axis-aligned depth-0 line), the C's view re-fit yields a zero `delta`, and the *next* frame's `((x - min) / delta) * width` divides by zero — in C that's `(int)inf`, undefined garbage the X window simply clips. Canvas would stroke `NaN` coords (a no-op, but sloppy), so the re-fit clamps `deltaX`/`deltaY` to `EPSILON` (1e-5), keeping the mapping finite and drawing a valid framed line. Without the guard ~1 frame in ~1400 produced a non-finite coord; with it, 0 across 65M coords.
- **`random_double` is continuous.** The C quantizes to an epsilon grid (`(random()%steps)*eps`); a plain uniform draw over the same `[base, limit)` range is equivalent for the geometry and avoids the integer-step bookkeeping. Angle/length ranges are unchanged.
- **EPSILON bail — slightly stricter recovery.** `self_similar_normalized`'s "did the realigned chain land back on `(x2,y2)`?" check (the C's commented-out assertion, replaced by jwz with a `return False`) is kept. This only fires for a numerically broken/degenerate generator, which is rare. *Minor deviation:* in the C the outer `self_similar` is `void` and **discards** that `False`, so on a bail the C keeps going with whatever partial extents it had (a few off-screen/garbage frames) until `draw_index` reaches `draw_iterations`; the port instead sets `draw_index = 0` and restarts a fresh fractal immediately. Benign (the bail is degenerate either way), and it avoids drawing garbage, but it is not a byte-for-byte reproduction of the C's "ignore and continue".
- **`devicePixelRatio`**: backing store sized in device px; line width is `max(1, dpr)`, matching the C's 1-device-px hairline (`XDrawLine`, default GC line width 0) scaled for retina. Plot-space math is resolution-independent, so geometry is unchanged. Keypress/`fps` handling dropped (the host owns keys and the meter).

## Palette (faithful)
The C builds its colormap **once** in `ccurve_init` via `make_color_loop` with anchors `(0,1,1)/(120,1,1)/(240,1,1)` — a full-saturation, full-value hue loop red->green->blue->red — and **never cycles or rebuilds it**. Because all three anchors are `s=1, v=1`, `make_color_loop` (= `make_color_path` over 3 points) collapses to a uniform hue sweep, which is *exactly* `hsl(h, 100%, 50%)` (hsl 100%/50% == hsv s=1/v=1). The port emits the same **256** `hsl()` entries (`MAXIMUM_COLOR_COUNT`), built once, not cycled, not user-configurable. A leaf's colour index is `floor(line_count * 256 / total_lines)` walking along the curve. `colormap.js` has no `make_color_loop`/`make_color_path` export, but the `s=v=1` special case makes this direct `hsl` computation byte-faithful, not an approximation, so no helper is required.

### Audit fix (2026-06-28)
The earlier port exposed an **invented `Colors` slider** (`ncolors`, default 64) the `.xml` never had, which banded the rainbow into 64 steps (vs the C's 256) and washed every hue at `55%` lightness. Removed: the palette is now the fixed 256-entry, full-saturation `make_color_loop` rainbow at `50%` lightness. The two real sliders were relabelled to the `.xml` verbatim (`Animation speed` / `Change image every` / `Density`, with the xml's low/high labels), and the non-faithful large-window `2*dpr` line-thickening was dropped for the C's true 1-px hairline.

## Correctness self-review
- **Termination / progression.** Instrumented the leaf-per-frame sequence: it reads exactly `1, sc, sc^2, ... , sc^(iters-1)` then drops to `1`, i.e. depth climbs one level per frame and resets cleanly at a new fractal — for seg=4 it ran depths 0..7 (max 16384 leaves), seg=3 depths 0..10 (max 59049), matching `floor(log(limit)/log(sc))`. 109 restarts over 400 driver ticks; no runaway recursion (depth is bounded by `iterations`, which is `>= 1`).
- **Every state path re-seeds what the next frame reads.** `draw_index==0` always runs `newFractal()` (picks generator, re-normalizes `points`, recomputes `iterations`, reallocates the per-depth scratch buffers) before drawing; the view box and endpoints persist across fractals exactly as in the C (only `reset()` restores them to the `ccurve_init` values). No "dead" state: a bailed fractal sets `draw_index = 0` and returns 0 so the next step starts fresh immediately.
- **No freeze.** Variable-delay loop caps the backlog at `nextDelay + 1000` (so a long linger still elapses) and `MAX_CATCHUP_STEPS = 4` (small, because each step can stroke tens of thousands of segments — we don't want a refocus to fire several full deep frames at once). `pause`/`resume` reset `lastTime` so there's no catch-up burst; `reinit` blanks the canvas and restores the init view for a clean fresh sequence.
- **No over-draw / no NaN.** Closure here isn't float-equality (there's no closed loop to detect) — the recursion is bounded by integer depth, so the figure-never-resets class of bug can't occur. The only numeric hazard was the zero-extent division, now guarded; 0 non-finite coords across 65M strokes.

## Config
Mirrors `hacks/config/ccurve.xml` exactly — only the three real resources, with their xml labels: `pause` (**Animation speed**, live, inverted, 0-5 s, Slow/Fast), `delay` (**Change image every**, live, 0-30 s, "0 seconds"/"30 seconds"), `limit` (**Density** — sets recursion depth, reinit, 3-300000, Low/High). The `showfps` boolean is the host's, not a param. There is **no** `ncolors`/Colors slider (it was invented; the C's palette is fixed — see Palette above). Non-live changes and "Reset to defaults" re-run `reset()` (fresh sequence from the init view). `r` (restart) also resets.
