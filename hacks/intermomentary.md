# intermomentary — port notes

Port of `intermomentary.c` (Mike Kershaw / "dragorn", 2004; a direct port of j.tarbell's "InterMomentary", complexification.net — a REAS collaboration with Robert Hodgin and William Ngan for the Whitney's ARTPORT; xscreensaver machinery by Jamie Zawinski) — a field of slowly drifting discs that lights up the **momentary intersection points** where their outlines cross, plus a swarm of points riding each disc's perimeter that flare into glowing orbs whenever they pass across a live intersection. **Sister hack to [[interaggregate]]** (same author family, same radical-line intersection math). See [[binaryring]] for the per-pixel-alpha BLIT idiom this follows and [[squiral]] for the shared module skeleton.

Original: <https://www.jwz.org/xscreensaver/> · source: `xscreensaver-6.15/hacks/intermomentary.c` (~605 lines)

## Algorithm
The whole image is a single-channel **brightness buffer** (the C's `off_alpha`, one `unsigned char` per pixel) that is **wiped at the start of every frame** — so unlike its sister this is the *instantaneous* view, not a lifelong aggregate (the .xml's "aggregate intersections" wording notwithstanding; `blank_img` zeroes the buffer each frame). Per frame, for each disc in `id` order:
1. **Move** it (`move_disc`): add velocity, wrap around the torus (offset by the radius so it slides fully off before reappearing), and creep the current radius `r` toward its destination `dr` by `0.1` px/frame. Discs are born with `r = frand(dr)/3` (≈ 1px and up) and grow.
2. **Draw intersections** with every *higher*-`id` disc (`render_disc`): two discs cross when centre distance `d` lies strictly between `|r1-r2|` and `r1+r2`; the two crossing points come from the standard radical-line construction (`a` along the centre line, `h` the half-chord perpendicular). Each crossing point is alpha-blended into the buffer **bright** (`trans_point(255, 0.75)`).
3. **Move + render the riders** (`moverender_rider`): each disc carries up to `maxRiders` "pixel riders" — points orbiting the perimeter at angle `t`, with a jittering, friction-braked angular velocity `vt`. At each rider's perimeter pixel: if the buffer there is *already* lit beyond a small threshold (`cv > 0.0006921`, i.e. a fresh intersection brightness sits here), it **flares into a 5×5 glowing orb** (`draw_glowpoint`) and recharges; otherwise it decays its own little `mycharge` and lays a very faint point.

So the bright intersection dots and the glowing orbs are the show: the orbs blink on only where a perimeter point happens to ride across a live crossing — "blinking dots interact with each other circularly." Processing discs in `id` order matters: a disc's riders can flare on the intersections drawn *earlier in the same frame* (its own with higher discs, plus any lower disc's with it). The buffer is finally colour-mapped through the C's bg→fg ramp (**black → yellow**) to the screen.

## Rendering approach
The C keeps an offscreen `unsigned char` alpha map, alpha-blends individual pixels into it (`trans_point`: `new = old + (src-old)·a`), and `XCopyArea`s the whole thing to the window each frame — the textbook BLIT path. So this mirrors [[binaryring]]: a persistent `Float32Array` brightness buffer (one channel, wiped each frame) blended per pixel, and a `Uint32` `ImageData` that the buffer is colour-mapped into via a 256-entry `ramp[]` and `putImageData` once per frame. Because the C draws each touched pixel to the screen via `get_pixel` *as it blends* but then copies the whole off-map anyway, the screen content is exactly `ramp[buffer]`, so baking the colour map once at frame end is equivalent and cheaper than per-blend screen writes. A frame is ~85 discs × (a handful of intersecting pairs + ≤40 riders) blends, then one full-buffer map — well within budget.

## Deviations from the C
- **Brightness buffer is `Float32`, not `unsigned char`.** The C truncates each blend back to an integer 0–255, which all but zeroes the faint rider points (`255·mycharge·0.5 ≈ 0.49 → 0`) so the bare circle outlines are essentially invisible between flares. Keeping the buffer in float preserves that subtle outline glow while the **bright** intersection path (`0.75·255 = 191`) and the rider **threshold** (`> 0.176`) behave identically — the glow trigger and the orbs are unchanged. This is the one intentional fidelity tweak (toward *slightly* more visible outlines); everything else is faithful.
- **Colour ramp baked at frame end**, not per-blend (see *Rendering approach*) — equivalent output, fewer writes.
- **`pscale` folded into `devicePixelRatio` (`S`).** The C bumps a `pscale` to 3 on >2560px (retina) displays to keep dots visible, and draws `pscale×pscale` rects while only blending single pixels into the buffer. Here the backing store is sized in *device* pixels and disc radii / growth rate / velocity offsets / the `5px` base are scaled by `S`, so density is dpr-correct and a "pixel" is already a device pixel — the buffer maps 1:1 to the screen (`pscale 1`). Net effect matches; the retina kludge is unnecessary. The one quantity that must **not** scale with `S` is the per-disc rider **count** (`numr = frand(r)/2.62`): the C computes it from the *unscaled* radius, so the port divides `S` back out (`frand(r/S)/2.62`). Otherwise retina (`S>1`) would roughly double the perimeter dots, over-populating each disc. Radius and velocity stay `S`-scaled.
- **Black → yellow ramp** built by the shared faithful `colormap.js` `makeColorRampRGB(0,0,0 → 60,1,1, 256, closed_p=False)` — the exact `make_color_ramp(bg=black → fg=yellow h=60 s=1 v=1, closed=False)` call from `intermomentary_init`. It is HSV-interpolated, so the midtones are warm **amber/gold** (e.g. index 128 ≈ `[127,95,63]`, index 192 ≈ `[191,155,47]`), tailing up to near-yellow `[254,250,0]`. Built once per init like the C and deterministic (no RNG). The earlier port flattened this to a literal `rgb(v,v,0)` lemon-yellow ramp, which does **not** reproduce `make_color_ramp` (it skips the amber/gold midtones) — now fixed. Kept faithful (yellow orbs on black) rather than swapped for a rainbow.
- **`delay` → `config.delay`** drives the rAF lag-accumulator instead of the C's returned `draw_delay`/`usleep`; identical pace at any refresh rate, with a catch-up cap. Default is the stock `30000 µs` (the slider maps 1:1 to the xml resource). The loop adds `OVERHEAD = 6900 µs` so `(delay + OVERHEAD)` reproduces the live binary's effective rate: measured against the live `-fps` overlay intermomentary runs **27.1 fps**, while the port at the stock delay ran ~33 fps (1.2× fast); `30000 + 6900 = 36900 µs → 27 fps`. A calibration, not a tuning knob — see the framerate-calibration note.
- **Added knobs** beyond the stock UI (the .xml exposes only frame rate, disc count, fps): `maxRadius` and `maxRiders`, which are real `--max-radius` / `--max-riders` options (C defaults 100 / 40), surfaced for parity with the other ports. The C's hard minimums (`numDiscs > 10`, `maxRadius > 30`, `maxRiders > 10`) become the slider `min`s (11 / 31 / 11) so no invalid config is reachable.
- **Edge wrap of the rider angle** uses floored modulo (`((t + vt + π) % 2π) - π` then a `+ 2π` fix-up), because JS `%` keeps the dividend's sign where C's `fmod` would too — but the subsequent `- π` can leave the value below `-π`, which the fix-up corrects. Verified the angle stays in `(-π, π]`.

## Correctness self-review
- **First frame is non-degenerate.** 85 discs are seeded across the canvas on an anti-collapsing ring (`0.4·W·cos`, `0.4·H·sin` offsets + a random in-quadrant jitter) with destination radii `5–105 px·S` and starting radii `frand(dr)/3`. Many overlap from frame one, so intersections and orbs appear immediately — no blank or off-screen start.
- **No closure/reset trap (the classic failure mode).** Nothing here *closes a figure* or relies on float-equality to terminate: discs drift and wrap forever, radii grow once toward `dr` and then sit (the `if (r < dr) r += …` is a one-way creep with no reset needed), and riders orbit perpetually. The buffer is unconditionally `fill(0)`'d every frame, so there is no accumulation that could saturate or "over-draw" — the screen can't freeze or fill in. The only state that resets is per-`reinit`/`resize` (fresh field).
- **Bounds + NaN guards.** `transPoint` is a no-op outside `[0,W)×[0,H)`; the intersection construction rejects disjoint (`d ≥ r1+r2`) and contained (`d ≤ |r1-r2|`) pairs before computing, the half-chord sqrt is clamped `max(0, r²-a²)` against round-off, and both crossing points are bounds-checked (the pair is skipped if either is off-screen, matching the C's `continue`). Rider points are bounds-checked before the buffer read. So no out-of-range index and no `NaN` brightness can reach the buffer.
- **Threshold semantics preserved.** The orb trigger `cv > 0.0006921` (buffer brightness `> ~0.176`) fires only where an intersection's `191` (or another orb's blend) already sits this frame; a lone faint rider point (`≈ 0.49`) is below it, so riders don't self-trigger — exactly the C's behaviour, and the reason the orbs *blink* rather than smear.
- **Pause/resume / reinit.** `pause()` cancels the rAF and parks `rafId = 0`; `resume()` resets `lastTime = 0` before re-arming so the banked `lag` can't fire a catch-up burst. `reinit()`/`resize()` rebuild the buffer + ImageData + ramp and re-seed the field cleanly with the current config (disc count / radius / rider count are `live: false`, so they re-seed; frame rate is `live: true` and applies instantly).
