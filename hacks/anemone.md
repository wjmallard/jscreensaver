# anemone — port notes

Port of `anemone.c` by Gabriel Finch (2001) — a sea-anemone of many tendrils rooted at one central base, each a wiggling multi-segment line that grows and withdraws while the whole cluster slowly swirls. One-line summary (verbatim from the xml): *"Wiggling tentacles."*

Original: <https://www.jwz.org/xscreensaver/> · source: `xscreensaver-6.15/hacks/anemone.c` (~457 lines) · config: `xscreensaver-6.15/hacks/config/anemone.xml`. See [[squiral]] for the shared module skeleton and [[nerverot]] for the wiggling-tendril vector idiom.

## Algorithm
Each of `arms` tendrils is a polyline built by a **doubly-integrated random walk**: a new point's velocity is the previous point's velocity plus a `{-1,0,1}` jolt per axis, and its position is the previous position advanced by that running velocity (`pos[m] = pos[m-1] + vel[m-1]`, `vel[m] = vel[m-1] + jolt`). All arms are rooted at the origin (screen centre).

Each frame:
- **withdraw pass** (`animateAnemone`): a shrinking arm (`growth < 0`) drops its tip point with probability `rate/25`; when its `growth` counter climbs back to 0 it flips to growing (`growth = RND(finpoints - numpt) + 1`).
- **grow / withdraw triggers** (`createPoints`): one `RND(withdraw)` roll per frame — `0` makes the whole anemone fully withdraw (and reverses the swirl), `<11` makes a partial synchronised withdraw, otherwise each arm with probability `rate/100` (when `growth > 0`) decrements `growth` and appends a point (capped at `finpoints-1`); when `growth` hits 0 it flips to withdrawing (`growth = -RND(finpoints) - 1`).
- **render** (`drawImage`): clear to black, then stroke each arm with a small per-frame `+/-2px` jitter (the "wiggle" — applied to the rendered coords only, never stored, so the skeleton stays stable), capping each far tip with a fat round dot (the C's degenerate `width*3` round-capped segment). `x`/`z` are projected through a slowly advancing `turn` angle (`screenX = cx*cos - cz*sin`), `y` maps straight to screen — so the cluster swirls about its vertical axis.

## Rendering
Genuinely line-shaped, so canvas **vector** ops — and exactly like `braid.js` / `nerverot.js` the segments are **bucketed by colour index into one `Path2D` per colour** (`<= ncolors` strokes/frame), plus a second per-colour `Path2D` of filled tip dots. Full repaint each frame (clear to black + redraw), matching the C's double-buffered clear-and-redraw (no trail).

## Deviations from the C
- **No XOR / persistence tricks needed** — the C already does a full `XFillRectangle` clear then redraws every frame, so the plain Canvas "clear + redraw all arms" is faithful (no trail to emulate).
- **devicePixelRatio scaling**: all geometry, jitter, and line widths are multiplied by `S = devicePixelRatio` so the anemone keeps a consistent CSS size on retina. The C instead bumps `width*=4; finpoints*=2; withdraw*=2` when a screen exceeds 2560px; that retina hack is **dropped** in favour of `S`-scaling (cleaner, same intent).
- **Initial seeding**: the C's init loop computes `x = 1 - RND(1001)/500` with C **integer** division, which collapses `x,y,z` to `{-1,0,1}`, and its `while (dist >= 1)` reject-loop only accepts `dist == 0` — so every arm in fact starts at the exact origin with zero velocity and `numpt == 1` (nothing drawn), blooming outward only over the first few seconds. We reproduce that origin/zero-velocity start, but then **pre-grow each arm to a random length** (using the same add-point walk) so frame 1 already shows a full, varied anemone instead of a bare centre. Documented per the brief's "full on frame 1" requirement.
- **Swirl-reversal quirk preserved**: the C flips `turndelta`'s sign *inside* the per-arm loop on a full-withdraw event, so it only nets a reversal when `arms` is odd (with the default 128 it never reverses). We replicate this exactly (flip `turnSign` per arm) rather than "fixing" it. Also added a `turn < 0` wrap (the C only wraps `turn >= 2*PI`) so a reversed swirl can't drift unbounded.
- **Tip dot** drawn as a filled circle of radius `1.5*width` (equivalent to the C's zero-length `width*3` round-capped line). `lineJoin` is `round` vs the C's `JoinBevel` — indistinguishable on thin lines.
- **Colours** (faithful): `make_smooth_colormap` of `colors+3` entries, ported via `colormap.js` `makeSmoothColormapRGB` — 2-5 HSV anchor hues interpolated into a smooth closed loop, so any given run is a harmonious two-to-five-hue ramp (NOT a full-spectrum rainbow). Built once per `init()`, exactly as the C builds it once in `anemone_init`; each arm picks a fixed random index (`aCurr->col = st->colors[random()%ncolors]`). An earlier revision drew a vivid full-saturation `hsl()` rainbow here — that has been fixed to the smooth colormap (verified against the live binary: both render a smooth 2-anchor loop, ours blue↔crimson, the live run green↔blue). `colors` is still exposed as a slider even though the xml GUI omits it: it is a real `-colors` command-line resource (default 20), not an invented knob.

## Correctness self-review
Traced the grow/withdraw state machine by hand and confirmed with a 20,000-step headless harness (128 arms): **`numpt` stays within `[1, finpoints-1]`** — never stuck at 0 (the `numpt > 1` guard on `pop`) and never unbounded (the `< finpoints-1` add cap, with `growth` still counting down to the withdraw flip at the cap). `growth` oscillates finitely in `[-finpoints, +finpoints]`; every withdraw state re-seeds a positive `growth` and every grow state re-seeds a negative one, so no arm gets stuck. No NaN/Inf coords. Arms random-walk from a fixed central base; the bulk stays near centre (typical tip excursion small; a rare outlier reached ~768 logical px over the whole run — the C is identical and simply clips off-screen). `pause`/`resume` reset `lastTime` so there's no catch-up burst; `reinit` clears to black and re-seeds for non-live changes (arms / finpoints / colors).

## Config
Units and defaults mirror `hacks/config/anemone.xml`: `delay` (µs/frame, **40000** = the xml stock, inverted "Frame rate" slider), `arms` (128), `finpoints` ("Tentacles", 64), `width` ("Thickness", 2), `withdraw` ("Withdraw frequency", 1200; higher = rarer), `turnspeed` (50). `delay`/`width`/`withdraw`/`turnspeed` are **live**; `arms`/`finpoints`/`colors` size the arm arrays / palette so they re-run `init()` via `reinit()`.

## Speed
`delay` is the xml stock **40000 µs** (the author's inter-frame sleep), not a by-eye value. The rAF loop paces at `(delay + OVERHEAD)`: `delay` is only a floor, and the live binary's real rate is lower because every frame fully clears and redraws all arms. `OVERHEAD` is the live-measured **9000 µs** (live 20.4 fps, Load 18.5%, clean at stock delay 40000 — very delay-bound, so the ~9000 proposal happened to match).
