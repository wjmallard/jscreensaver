# swirl — port notes

Port of `swirl.c` by M. Dobie & R. Taylor (1994; turned standalone by jwz, 1997) — flowing, swirly patterns. A handful of spiral "knots" define a per-pixel field that is **revealed coarse-to-fine by a centre-out square spiral**, held static, then regenerated with fresh knots and a fresh colourmap.

Original: <https://www.jwz.org/xscreensaver/> · source: `xscreensaver-6.15/hacks/swirl.c` (~1446 lines, mostly X11 colormap plumbing). See [[imsmap]] for the same build-reveal-hold-regenerate lifecycle, [[metaballs]]/[[marbling]] for the offscreen-field + `drawImage`-upscale (retina-downscale) idiom.

## What it is
1. **Knots.** Each swirl scatters `n_knots = rand(count/2) + count + 1` knots (default `count = 5` → 6-8 knots). Each knot has a random position, a random mass `m ∈ [1, 5]` that is negated ~50% of the time, and a random spiral **type**: `ORBIT`, `WHEEL`, `RAY`, or `HOOK`. (The C's `ALL` mode pointedly omits `PICASSO`; we mirror that — `PICASSO` exists in the code but is never seeded.)
2. **Field.** Every pixel's value is the signed sum over all knots of that knot's contribution at the pixel (`do_point`): a `1/(1+dist)` falloff for orbit, an `atan2`-based angle term for wheel/ray/hook, etc. `+ve` masses add, `-ve` masses subtract. The integer total is folded into `[0, ncolors)` (the C's asymmetric `%dcolours+2` / `dcolours-(|v|%(dcolours-1))` wrap, kept verbatim). This `do_point` math is a faithful transcription and is sampled lazily by the reveal — never an ad-hoc rainbow/ramp.
3. **Reveal = centre-out square spiral, coarse → fine.** `draw_swirl` paints the field with a growing square spiral from the centre. `resolution` starts coarse (`MIN_RES + 1 = 8`) and decrements down to `max_resolution` (`MAX_RES = 1`, or `2` in two-plane mode). For each resolution the block step is `r = 1 << (resolution - 1)` (128, 64, …, 1); each frame paints `BATCH_DRAW = 100` "points" in the square-spiral order (`next_point`: arms grow 1,1,2,2,3,3,…), each point an `r × r` block of the field's colour at that cell (`draw_point` → `do_point`). So the picture **fades in from chunky blocks to full resolution**, redrawn finer at each resolution.
4. **Static hold — NO colour cycling.** Once the finest resolution finishes, the image sits **perfectly still** for `RESTART = 2500` frames. `swirl.c` only rotates its colourmap `if (mi->writable_p)` — i.e. on a **PseudoColor** (writable-colourmap) X visual. A browser canvas is **TrueColor** (`writable_p` is false), so the C does **not** cycle there; the revealed image is static (confirmed against the live binary: static + coarse→fine).
5. **Regeneration.** After the hold, the C re-seeds everything (new knots, new colourmap via `make_smooth_colormap`) and replays the coarse→fine reveal — a brand-new pattern. Lifecycle: build field → reveal coarse→fine → hold → regenerate.

## Shared skeleton (inherited)
Standalone ES module exporting `title`/`info`/`start(canvas) → { stop, pause, resume, reinit, config, params }`; rAF **lag-accumulator** loop (fixed timestep paced by `config.delay` µs, one `draw_swirl`/step per delay); `devicePixelRatio` folded in (backing store in device px, field at logical px); shared config box via `params`.

## Rendering (per-pixel field, retina downscale MANDATORY)
The field is a dense per-pixel thing with a `sqrt` + `atan2` **per knot per pixel**, so it uses the **blit path** + the **retina-downscale** idiom: a persistent `Uint32` grid buffer at **logical** resolution (`canvas px / dpr`) is `ctx.drawImage`-upscaled (bilinear) to the device-px canvas. Computing at device resolution on retina would be seconds per frame.

- **Cell cap.** The logical grid is capped at `MAX_CELLS = 360000` (~758×474). On a hi-DPI laptop (dpr 2) the cap engages and the field upscales ~3-4× — swirl's fine structure (ray lobes, wheel spokes) softens a little. On a 1× display at/under the cap it is pixel-exact.
- **No upfront hitch.** `do_point` is evaluated **lazily, only at the spiral's block corners** (exactly as the C's `draw_point` calls `do_point`), so the heavy field cost is spread naturally across the reveal frames; the **per-frame** cost is `BATCH_DRAW` block fills + one `putImageData`/`drawImage`. The static hold paints nothing (the blit is skipped), so it costs nothing and stays byte-identical.

## Deviations from the C
- **Palette = faithful `make_smooth_colormap` (`colormap.js`).** The colourmap is the exact `utils/colors.c` `make_smooth_colormap` — 2-5 random HSV anchors with min-separation + min-avg-saturation/value retries, frequently muted/pastel and **re-rolled fresh on every regeneration** (matching the C's cadence: the framework's `color_scheme_smooth`/`#define SMOOTH_COLORS` builds it at startup, and `draw_swirl` re-runs `make_smooth_colormap` on every `RESTART`). (An earlier port used a fixed vivid 3-hue HSL triangle modelled on `basic_map` / the 13-entry `basic_colours` table — but that code lives under `#ifndef STANDALONE` and is **dead** in this build, so it was never the right reference.)
- **two-plane mode.** Used ~30% of swirls (`TWO_PLANE_PCNT`), where each knot has two types drawn interleaved; `max_resolution = 2` (chunkier finest), faithfully replicated, including the `draw_point` split of each block into four `r/2` sub-blocks. The C interleaves by toggling `first_plane` on every `do_point` call; we interleave **per pixel on a `(x+y)` checkerboard** (a minor, accepted approximation of the plane assignment).
- **Faithful edge-parity remnant.** Because `draw_point`'s bounds guard, the centre/step, and the spiral termination are transcribed verbatim, two-plane swirls leave a few cells near one corner unreached by the finest pass; they stay the window background (black), exactly as in the C (which `XClearWindow`s to black and never repaints the background). ~0–0.6% of cells, always on the edge — not a coverage gap.
- **`ncolors` clamped to ≥ 2** (the xml allows `low = 1`). The field-fold divides by `(ncolors - 1)`, so a 1-colour map is disallowed; the slider min is 2. The default 200 uses the full palette.
- **Removed: colour cycling + its invented controls.** The previous port animated a continuous user-space colourmap rotation (with `cyclespeed`/`duration` sliders) and revealed the field as horizontal bands over `BUILD_FRAMES`. The C does neither on TrueColor — both were unfaithful and have been removed in favour of the centre-out square-spiral reveal + static hold above.

## Correctness self-review
Verified with a headless node harness that mocks a minimal DOM and drives the **real** module's rAF loop, capturing the offscreen grid `ImageData` it blits (`scratchpad/swirl-smoke.mjs`, 1 step/frame, plus a 12-swirl coverage sweep `swirl-coverage.mjs`):
- **(a) Progressive reveal.** The black grid fills in over the reveal (black-cell count `4800 → … → ~0` across the frames; many distinct frame checksums) — coarse→fine, not instant. Single-plane swirls reveal over ~80 frames (finest `r = 1`); two-plane over ~27 (finest `r = 2`).
- **(b) Static hold = byte-identical.** Across the entire `RESTART`-length hold (~2502 frames) the grid buffer's checksum is constant and a direct byte-compare of two hold frames differs in **0** bytes — proof there is no colour cycling and the field is static within a swirl.
- **(c) Regeneration.** After the hold the field's checksum changes (new knots + a fresh `make_smooth_colormap`), and the next hold's field differs from the previous — successive swirls differ in geometry *and* colour.
- **Coverage.** Across 12 random swirls the residual unpainted (black) cells are 0–28 of 4800 (≤ 0.58%), **always on the grid edge** — the faithful two-plane parity remnant, never a contiguous interior gap. Final fields span the full palette (≈ 200 / 200 distinct colours); every pixel is opaque (no out-of-range palette index).
- **Lifecycle.** `pause()`/`resume()` (resets `lastTime` so no catch-up burst), `reinit()` (rebuilds grid/knots/palette via `init()`), `stop()` (cancels rAF + removes the resize listener) all run clean.
- **Live vs non-live.** `delay` is read every step → `live: true`; `count`/`ncolors` resize the knot arrays / palette → `live: false` (re-run `init()` via `reinit()`).

**Worth a browser spot-check (new hack):** the *aesthetics* on a real retina display — whether the `MAX_CELLS` cap leaves enough crispness in the ray/wheel detail (vs. too soft), and whether the coarse→fine reveal pace at the default `delay` reads well (the finest `r = 1` pass is a slow pixel-by-pixel fill, faithful to stock; the `delay` slider tunes it). If it looks too soft, raise `MAX_CELLS`.

## Config
Mirrors `hacks/swirl.xml` exactly (only its real, animatable resources): `delay` (µs/frame, default 10000, `invert: true` "Frame rate" slider), `count` → **Count** (knots, 0-20, default 5), `ncolors` → **Number of colors** (clamped 2-255, default 200). `showfps` is a framework control, not surfaced. There is **no** cycle-speed or duration control — the C does not cycle, and the hold is the hardcoded `RESTART`.

**Framerate calibration (`OVERHEAD = 6556`).** The stock `delay = 10000 µs` is only a sleep floor; the live binary's real rate is lower (delay + framework overhead — see the framerate-calibration note). The live swirl measures **60.4 fps**, while the port at the stock delay ran 100 steps/sec (1.66× fast). The loop adds `OVERHEAD`: `10000 + 6556 = 16556 µs → 60.4 steps/sec`, matching the live binary. The `RESTART` hold is **frame-counted**, so it scales with `(delay + OVERHEAD)` exactly as the live binary's; the `delay` slider still maps 1:1 to the xml resource.

**Local dev:** ES-module `import`s need a real server — `python3 -m http.server` in the repo, then <http://localhost:8000/#swirl>. `file://` double-click fails (CORS on the `null` origin); GitHub Pages serves over http, so production is fine.
