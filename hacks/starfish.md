# starfish — port notes

Port of `starfish.c` by Jamie Zawinski (1997) — "Undulating, throbbing, star-like patterns pulsate, rotate, and turn inside out." One big radial-spline blob (a starfish) whose arm radii oscillate while the whole shape slowly spins; each frame fills the band the outline just swept, in the next colour of a slowly-cycling colourmap.

Original: <https://www.jwz.org/xscreensaver/> · source: `xscreensaver-6.15/hacks/starfish.c` (563 lines) + `xscreensaver-6.15/utils/spline.c` (the closed-spline maths) + `xscreensaver-6.15/utils/colors.c` (make_smooth_colormap / make_uniform_colormap). Demo video: <https://www.youtube.com/watch?v=atwc7IJHuLo>.

## Algorithm
A starfish is `npoints` control points spaced evenly around a centre. Every `skip`-th point is a **valley** (radius 0) and the rest are **peaks** (radius `size`); the number of arms is `npoints / skip`. Each frame:
- **throb** — each control point is placed at `(x + r·cos(i·frac + theta), y + r·sin(...))` using its current radius, then its radius steps toward the opposite extreme by `elasticity` (px/frame), with an easing factor that slows it near `min_r`/`max_r` (fastest in the middle) and reverses sign at the limits. So every radius oscillates between `min_r` (5px) and `max_r` (`size`), each with its own phase — the shape continuously morphs between a sharp star and a round blob, and "turns inside out" when peaks shrink while valleys grow.
- **spin** — `theta` advances by `rotv`, `rotv` accelerates by `rota`, `rota` bounces at `±rot_max` and is occasionally perturbed/reversed, so the rotation drifts, slows, stops and reverses organically.
- **draw** (`draw_starfish`) — the control points are joined into one **closed** smooth spline. The C then fills, **with the EvenOddRule**, the polygon formed by concatenating THIS frame's outline with the PREVIOUS frame's outline — i.e. the thin **band** the shape just swept between consecutive frames — in `colors[fg_index]`. `fg_index` advances one colourmap entry per frame.

The two render modes (the `mode` resource → `st->blob_p`):
- **Color gradients** (`zoom`, `blob_p == False`): the window is cleared **exactly once**, at startup, to `colors[0]`; thereafter it is **never cleared**, so the per-frame bands **accumulate** into concentric colour rings — the signature "gradient" look. This persists across shape re-rolls.
- **Pulsating blob** (`blob`, `blob_p == True`): the window is **cleared every frame** before the band fill, leaving a single writhing shape on black. blob mode also runs `elasticity *= 3`, `rotv *= 3` and `delay *= 3`, so each frame's band is much thicker and the motion chunkier (same net real-time pace).
- `random` is 1/3 blob, decided once at init.

Independently of `blob_p`, each freshly-rolled shape picks a *deformation* mode (the C's per-shape `s->mode`, pulse vs zoom): "zoom" pins the valleys at the centre (sharp permanent arms), "pulse" throbs everything. Every `duration` seconds the shape is freed and re-rolled (new arm count / skip / spin / size); 1-in-10 of those also rebuilds the colourmap. A re-roll never clears the screen (zoom keeps accumulating).

The control points become a smooth curve via `compute_closed_spline()` → `calc_section()`, the standard **uniform cubic B-spline → Bezier** conversion wrapped around the control array. For section `i` (control `i` → `i+1`) the Bezier points are `p0=(c[i-1]+4c[i]+c[i+1])/6`, `p1=(2c[i]+c[i+1])/3`, `p2=(c[i]+2c[i+1])/3`, `p3=(c[i]+4c[i+1]+c[i+2])/6`, and `p3` of section `i` equals `p0` of section `i+1`, so the curve is C2-continuous and closes seamlessly.

## What this port keeps faithfully
- **Geometry & dynamics**: `makeStarfish`, `throb`, `spin` are a direct transcription — same `skips`/`sizes` tables, same `npoints = skip · multiplier` (with the four largest multipliers dropped when `skip > 3`), same valley/peak radius seeding, same elasticity easing and limit-reversal, same bell-curve random elasticity (`0..15`, avg 7.5) / rotation (`0..12°`, avg 6°) / occasional shrink, same spin acceleration/stop/reverse bookkeeping (sign-of-theta and all), and the **blob `×3` on elasticity / rotv / delay** (with `rot_max` taken after the `×3`, as the C orders it).
- **Render = the swept band, EvenOddRule, accumulating in zoom.** Each frame fills the even-odd region between this frame's outline and the previous frame's outline in `palette[fgIndex]`. zoom mode clears only once (to `palette[0]`) and then accumulates concentric rings; blob mode clears each frame. This replaces the earlier port's whole-shape solid/radial-gradient fill (a flattening — see Deviations).
- **`fg_index` colour cycling**: one colourmap **entry** per frame (`fgIndex = (fgIndex+1) % ncolors`); the fill is a single flat colour, not a gradient. The first frame after any (re)seed has no `prev`, so it only records the outline and draws nothing — matching the C's `if (s->prev)` guard — while `fg_index` still advances.
- **Colourmap = `make_smooth_colormap` (2/3) or `make_uniform_colormap` (1/3)**, via the faithful `colormap.js` helpers (NOT a vivid HSL rainbow). The 2/3 case is `makeSmoothColormapRGB(ncolors)`; the 1/3 case is `make_uniform_colormap`, which in the C is exactly `make_color_ramp(0, S, V, 359, S, V, closed_p=False)` with `S, V` each random in `(random()%34 + 66)/100` ≈ 0.66–0.99 — reproduced here as `makeColorRampRGB(0, sat, val, 359, sat, val, ncolors, false)`. The map is rebuilt **only** at init and on the 1/10 duration re-rolls, exactly as `reset_starfish` does; it is **cycled in place** the rest of the time.
- **The closed spline is exact**: the B-spline→Bezier formula above is `calc_section()` reduced; canvas draws the Beziers natively, so it stays smooth at any DPI with no integer-pixel polygon and no adaptive subdivision. The wrap uses `(i±k+n)%n` on every index, so there is no off-by-one kink at the seam.
- **EvenOddRule**: kept (`ctx.fill(band, 'evenodd')`), so self-crossings when the star turns inside out read as holes, exactly like the C's `gcv.fill_rule = EvenOddRule`.
- **Re-roll cadence**: every `duration` seconds a fresh shape is built; 1/10 of those also rebuild the colourmap (`reset` vs `rerollShape`), matching `starfish_draw`'s duration branch and its `random()%10` colour re-roll. `fg_index` continues across a 9/10 shape re-roll and resets to 0 only on a colourmap re-roll (as `reset_starfish` does).
- **Defaults/labels** from `starfish.xml`: `mode` (Random / Color gradients / Pulsating blob), `delay` 10000 (0–100000, inverted), `duration` 30 (1–60), `thickness` 0=random (0–150), `ncolors` 200 (2–255). No invented sliders; `delay2` (set-but-unused in the C) and the framework `showfps` are dropped.

## Deviations from the C
- **Even-odd band as two closed sub-paths vs one concatenated polygon.** The C builds a single polygon `[current_points ++ prev_points]` and even-odd-fills it. The port appends both closed outlines into one `Path2D` and fills it even-odd, which yields the identical region (the symmetric difference of the two outline interiors) apart from a measure-zero "connector" sliver the C's two joining edges produce. Visually identical.
- **Native Beziers vs a flattened polyline.** `compute_closed_spline` flattens each Bezier to line segments (`add_bezier_arc`, SMOOTHNESS≈1px); the port draws the true Bezier via `bezierCurveTo`. Sub-pixel identical, smoother at high DPI.
- **`duration` is accumulated sim time** (`delay` per step, `×3` in blob) rather than wall-clock `time()`, so it's frame-rate independent and pauses with the loop.
- **devicePixelRatio**: `size`/centre come from the device-px backing store and `elasticity`/`min_r` are scaled by `S`, so the shape and its throb keep the same proportions on retina (the C has no DPI concept). Counts/frequencies (npoints, skip, rotv, fg cadence) are NOT dpr-scaled.
- **Resize clears the canvas.** Setting `canvas.width` zeroes the backing store, so a resize starts a fresh accumulation in zoom mode; the C's `starfish_reshape` keeps the old pixels. Unavoidable platform difference, only visible at the moment of a resize.
- **`ncolors ≤ 2` (mono) not implemented.** The C treats `ncolors ≤ 2` as monochrome (black/white foreground toggle). The port clamps `ncolors` to ≥ 2 (the xml allows 1) and just builds a small smooth/uniform colourmap there. Rare edge; the default is 200.
- **`thickness` labelled "Thickness".** The C maps the `thickness` resource to `elasticity` (radial velocity); the xml's display label is the legacy "Lines / Thin / Thick". We surface it under its resource class name "Thickness" with thin/thick hints (`0` = random, matching the C). It sizes the motion, so it re-seeds (`live: false`).

## Frame-rate calibration (`OVERHEAD`)
The loop emits one step per `(delay + OVERHEAD)` of **wall-clock** time (the rAF lag-accumulator banks real elapsed `now - lastTime`), so the rate is machine-independent: the port draws the same number of bands per real second on any CPU, and a faster machine does not speed it up.

`delay` alone (the xml default 10000 µs) implies 100 steps/sec, but a 10 ms `usleep` is only a *floor* — the C pays a real per-frame cost (sparse-vector draw + the display's vsync/compositor cadence) on top of it. Measured against the live binary with the framework's `-fps` overlay: **55.6 fps, Load 44.4 %** in zoom mode at the default delay (i.e. NOT CPU-bound — idle 55 % of each frame; the rate is set by the delay floor plus a ~vsync-class cadence, not by horsepower). So the real per-frame period is ≈ 18 ms, and `OVERHEAD = 8000 µs` reproduces it exactly: `1e6 / (10000 + 8000) = 55.6`. The port re-measured at **55.5 steps/sec** after the fix, matching the live.

Without this the port laid bands ~1.8× too fast, so the zoom-mode rings washed out in half the wall-clock time (the live washes too — bold by ~8 s, muted by ~16 s, faded by ~24 s — just half as fast; verified by capturing the live's own zoom trajectory). `OVERHEAD` is added **once per frame, after the blob `×3`**, matching the C's real period `delay·[3] + overhead`; this is the same calibration the sibling geometry ports `dangerball`/`cubicgrid` use (theirs is ~37500 µs — a GL frame costs far more than a 2D vector draw, so the constant is per-hack). The fixed-`OVERHEAD` value pins every visitor to ~55.6 regardless of their display refresh; a display-adaptive vsync-lock was considered and declined (it would be a project-wide change and would disable the `delay` slider). See the `framerate-calibration` memory.

## STRUCTURAL REBUILD — live-verified
The render/colour core was rebuilt from scratch against `draw_starfish` + `reset_starfish` + `run_starfish` (the previous port had flattened it into a vivid-HSL whole-shape fill with a continuous hue rotation; see below). It is a line-for-line transcription of the C pipeline (even-odd band between consecutive outlines, accumulate-in-zoom / clear-in-blob, `make_smooth`/`make_uniform` colourmap cycled one entry per frame, blob `×3`). **Verified against the live binary (2026-06-27):** the zoom-mode wash trajectory and rate now track the original after the `OVERHEAD` calibration above (port 55.5 vs live 55.6 steps/sec; user confirmed the pace by eye). Note the radial sunburst streaks that appear on some runs are the faithful result of a re-rolled `defZoom` shape (valleys pinned at centre → sharp arms → thin radial slivers between consecutive outlines), present in the C too, not a port artifact.

### What the previous port did wrong (now fixed)
- Filled the **whole current shape** every frame (solid hue in blob, a **radial rainbow gradient** in zoom) and **cleared every frame in both modes** — so the zoom-mode band accumulation that produces the concentric "colour gradients" was entirely absent. Replaced with the faithful even-odd swept-band fill + zoom accumulation.
- Used a **vivid full HSL rainbow** (`hsl(h,100%,55%)`) and a continuous hue rotation. The C uses `make_smooth_colormap` (often muted/pastel) or `make_uniform_colormap`, indexed one entry per frame. Replaced with `colormap.js`.
- **Dropped the blob `×3`** on elasticity/rotv/delay (the old notes argued delay×3 cancels). It does not cancel for the *look*: the `×3` on elasticity/rotv sets the per-frame band thickness and the chunkiness, independent of the frame rate. Restored.
- Mis-described the C's accumulation as a "GXxor-style trick". It is a **plain GXcopy fill with EvenOddRule** on a never-cleared window — no XOR anywhere in `starfish.c`.

## Correctness self-review
- **No freeze / no runaway radii.** Each radius oscillates strictly within `[min_r, max_r]`: the easing factor is bounded `0.1..1.0`, and the limit test flips the sign at each end. A headless harness (700+ frames in each of zoom/blob/random, `duration = 1s` to exercise re-rolls) ran with no exceptions and no NaN/divergence. `max_r ≤ min_r` is guarded.
- **Render behaviour verified headless.** With stubbed `Path2D`/ctx: zoom does **zero** clears during the run (accumulates) while blob clears once per filled frame; every fill uses the `'evenodd'` rule; fill colours are muted `rgb(...)` colourmap entries advancing one step per frame (not HSL, not a gradient); `createRadialGradient` is never called.
- **blob `×3` verified**: over equal wall-time, zoom ran ~3× as many steps as blob (delay×3), confirming the blob delay multiplier.
- **Closed spline never kinks.** Every section index uses modular wrap, and `p3(i) == p0(i+1)` for all `i` including the `n-1 → 0` seam.
- **Re-seed is clean.** `makeStarfish` rebuilds every field the next `throb`/`spin`/`draw` reads; `reset` (init / 1-in-10 re-roll) rebuilds the colourmap + `fgIndex = 0`, `rerollShape` (9-in-10) keeps the colourmap and the cycling `fgIndex`; both clear `prev` so the new shape's first frame just records.
- **Colour cadence can't divide by zero.** `ncolors` is clamped to `[2, 255]`; `fgIndex` wraps mod `ncolors`.
- **rAF loop**: shared lag-accumulator, `MAX_CATCHUP_STEPS = 8`; `delayMs = 0` (max frame rate) is bounded by the step counter. `pause()` nulls `rafId`; `resume()` resets `lastTime`; `reinit()` re-runs `init()` (rebuild + clear).
- **Live vs reinit**: `delay`, `duration` are live (read each step); `mode`, `thickness`, `ncolors` size the shape / palette, so they re-run `init()` via `reinit()`.
